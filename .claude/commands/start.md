# /start — Швидкий старт нової сесії

**Коли використовувати:** перша команда у кожному новому чаті. Один виклик — і Claude готовий до роботи з повним контекстом проекту.

**Без аргументів.** Просто `/start`.

## Крок 1 — Прочитати обов'язкові файли (у цьому порядку)

**СПОЧАТКУ — ПРАВИЛА ВЗАЄМОДІЇ (обов'язково кожну сесію):**

1. **`CLAUDE.md`** — перечитати УВАЖНО секції "⚠️ ОБОВ'ЯЗКОВО В КОЖНОМУ ПОВІДОМЛЕННІ" (пояснення в дужках) і "📏 РОЗМІР ВІДПОВІДЕЙ" (середній розмір, не простирадла). Ці правила Claude систематично порушує між сесіями — треба свіжити. Хоча файл автоматично в контексті як project instructions, свідоме перечитання потрібне щоб правила були "гарячими"
2. **`РОМАН_ПРОФІЛЬ.md`** — хто такий Роман, як з ним комунікувати, стиль, заборонені кольори. Читати завжди — а не тільки при першій сесії — бо Claude забуває між чатами

**ПОТІМ — СТАН ПРОЕКТУ:**

3. **`START_HERE.md`** (у корені репо) — карта проекту, файлова структура, список скілів, перший орієнтир
4. **`_ai-tools/SESSION_STATE.md`** — поточний стан: версія, гілка, що робили в останніх сесіях, **відкрите на майбутнє**. Особлива увага до секції "⚠️ ДЛЯ НОВОГО ЧАТУ — найважливіше" на початку файлу
5. **`NEVERMIND_BUGS.md`** (у корені репо, НЕ в `_ai-tools/`) — відкриті баги з пріоритетами (🔴 критичні, 🟡 середні, 🟢 дрібні)
6. **`ROADMAP.md`** — **СТРАТЕГІЧНИЙ ПЛАН** (консолідовано з `FEATURES_ROADMAP.md` + `SESSION_STATE.md` → ROADMAP 13.04.2026). Дорожня карта фаз розвитку: `🚀 Active`, `📋 Next` (7 послідовних блоків робіт), `✅ Done`, `💡 Ideas`, `🚫 Rejected`, `🔒 After Supabase`. **Без цього файлу Claude не розуміє куди йде проект і починає вигадувати "власні" рішення які не відповідають концепції.** Читати щоб бачити: (а) що вже зроблено, (б) що в планах ДО Supabase по блоках, (в) що ПІСЛЯ Supabase, (г) загальну візію Jarvis як Reasoning Engine. Повні обговорення Grok/Gemini/DeepSeek/Perplexity + анти-патерни + архітектурні принципи — в архіві `_archive/FEATURES_ROADMAP.md`.
7. **`CONCEPTS_ACTIVE.md`** — реалізовані концепції вкладок (Finance, Evening, Me, Projects, Gamification). Дає зрозуміти ПОТОЧНИЙ функціонал. Читати швидко — достатньо зрозуміти які концепції активні

**🔥 ОСТАННІМ — `_ai-tools/HOT_RULES.md`:** компактний файл з 8 найболючішими правилами які Claude систематично порушує (пояснення в дужках, опис UI вигляду а не коду, розмір, швидкий діалог, без «Роби», глибина, CACHE_NAME, skeleton+Edit). Читати **в самому кінці** — щоб правила залишились у свіжій памʼяті ПЕРЕД першою реплікою Роману. Це не дубль `CLAUDE.md` — стислий гарячий мінімум з прикладами неправильно/правильно.

**Винятки — НЕ читати:** `_archive/*` (історичні документи), `bundle.js` (згенерований), `_ai-tools/REFACTORING_PLAN.md` (історична довідка, рефакторинг завершено), `NEVERMIND_LOGIC.md` (базова концепція — timeless, читати тільки за потреби).

## Крок 2.5 — 🔍 Перевірити кількість активних блоків у SESSION_STATE

**Чому тут:** правило `/finish` Phase 0 «архівація найстаршого блоку при ≤2 активних» працює тільки коли контекст <75%. На `/finish` контекст вже забитий — правило систематично пропускається. Зафіксована регресія 3 сесії поспіль (C8uQD 27.04 → rKQPT 02.05 → bOqdI 02.05) → 4 активних блоки замість ≤2. Тригер переноситься на `/start` де контекст свіжий.

**Виконати:**
```bash
grep -cE "^(<details>|## ⚠️ ДЛЯ НОВОГО ЧАТУ|^---$).*$|^📦 .* \(" _ai-tools/SESSION_STATE.md 2>/dev/null
grep -c "^<details>" _ai-tools/SESSION_STATE.md
```

**Дія:**
- Якщо `<details>` блоків **>1** (тобто разом з активним блоком зверху >2 активних) — у відповіді Роману **окремою стрічкою**: «⚠️ SESSION_STATE: N активних блоків (норма ≤2). Спочатку архівувати найстарші у `_archive/SESSION_STATE_archive.md`, потім решта роботи.»
- Якщо ≤1 — нічого не писати, переходити до Кроку 3.

## Крок 3 — 🎓 Самотест по правилах (ОБОВʼЯЗКОВО ПЕРЕД ВІДПОВІДДЮ)

**Виконати:**
```bash
bash .claude/hooks/start-self-test.sh
```

Хук покаже 3 питання про найболючіші правила (пояснення в дужках, UI = вигляд а не код, без «Роби» + глибина).

**Дія:** перед першою реплікою Роману — написати у чат блок `📋 Самотест:` з відповідями на 3 питання **своїми словами** на конкретному прикладі. НЕ цитувати правила — переформульовувати. Якщо не можу сформулювати — повернутись читати `_ai-tools/HOT_RULES.md`.

**Чому це треба:** Claude систематично читає правила пасивно (очі по тексту, мозок пропускає). Активна згадка через відповідь на питання своїми словами — переводить пасивне сканування у засвоєння. Без цього між сесіями правила знову розпорошуються.

**Після самотесту** — переходити до Кроку 4 (відповідь Роману).

## Крок 4 — Сформувати коротку відповідь Роману

Написати **4-6 рядків максимум** українською:

1. **Одне речення статусу:** "Прочитав. Проект: NeverMind v[версія], гілка [назва], остання робота — [що робили]."
2. **Найкритичніший баг** (якщо є в `NEVERMIND_BUGS.md` з 🔴 пріоритетом) — одне речення
3. **3-5 варіантів що робити далі** — коротко, нумерованим списком. Брати з `ROADMAP.md` секції `🚀 Active` (поточний пріоритет) або `📋 Next` (послідовні блоки робіт), або з відкритих багів у `NEVERMIND_BUGS.md`. Наприклад:
   - `/fix B-XX` — виправити конкретний баг
   - Верифікація старих багів проти реального коду
   - Нова функція яку обговорювали
   - Доопрацювання інструментів (tools — інструменти/скіли)

4. **Запитання:** "Що робимо сьогодні?"
