# /gemini — Запит другої думки від Gemini (ручна вставка в aistudio.google.com)

**Аргумент:** питання до Gemini (необов'язково). Приклади:
- `/gemini перевір мої зміни перед деплоєм`
- `/gemini я не знаю як краще зробити OWL reaction, запропонуй архітектуру`
- `/gemini` (без аргументу — тоді спитати Романа що він хоче від Gemini)

## Як це працює (концепція)

Це **не автоматичний MCP**. Це ручний workflow:
1. Claude збирає повний контекст проекту + git diff + резюме обговорення у чаті + питання Романа
2. Виводить усе **одним код-блоком у чаті** (чотири backticks навколо)
3. Роман тапає нативну кнопку Copy над код-блоком → відкриває https://aistudio.google.com або застосунок Gemini на iPhone → вставляє → читає відповідь Gemini
4. Роман повертає відповідь Gemini в чат → Claude реагує (погоджується або аргументує)

Інфраструктура нульова. Ключі ніде не засвічуються. Gemini 3 Pro має 1M+ контекст — вміщає весь NeverMind.

**Оновлено у сесії nudNp 24.04.2026:** попередній варіант (HTML-файл у корені репо з кнопкою Копіювати + пуш на прод) — скасований. Роман: «Нахуя мені по якихось силках переходити. Просто блок з текстом в нашому чаті з кнопкою копіювати». Claude Code Web має нативну кнопку Copy над кожним код-блоком — один тап і текст у буфері.

## Крок 1 — Отримати питання від Романа

Якщо після `/gemini` є текст — це питання, іди до кроку 2.

Якщо аргумент порожній — запитати:
> **Що ти хочеш від Gemini?** (наприклад: "перевір перед деплоєм", "запропонуй архітектуру для X", "знайди баги в моєму підході до Y")

Чекати відповідь Романа перед переходом до кроку 2.

## Крок 2 — Зібрати контекст з файлів

Виконати по порядку:

1. **Прочитати `CLAUDE.md`** (правила проекту — повний файл)
2. **Прочитати `docs/DO_NOT_TOUCH.md`** (священні корови — що не чіпати без обговорення)
3. **Прочитати `lessons.md`** (патерни / анти-патерни / варіанти що не спрацювали — щоб Gemini не пропонував знову те що вже відкинули)
4. **Прочитати `_ai-tools/SESSION_STATE.md` — тільки блок поточної/останньої сесії** (`## 🔧 Поточна сесія ...`). НЕ передавати весь журнал минулих сесій — це шум.
5. **`git branch --show-current`** — дізнатись поточну гілку
6. **`git diff main...HEAD`** — зміни від main до HEAD поточної гілки
   - Якщо diff порожній — пропустити, не включати в промпт
7. **`git diff main...HEAD --name-only`** — список змінених файлів
   - Для кожного файлу зі списку — **прочитати повністю** (Read tool). Винятки:
     - `bundle.js` — НЕ читати (згенерований, не потрібен Gemini)
     - `package-lock.json` — НЕ читати (занадто великий)
     - Бінарні файли (зображення, шрифти) — НЕ читати

## Крок 3 — Сформувати резюме обговорення з чату

**Це ключовий блок — «труднощі реалізації» за формулюванням Романа.** Перед збиранням промпту Claude сам формулює резюме з останніх повідомлень у ПОТОЧНОМУ чаті:

- **Яку проблему / ідею / концепцію обговорюємо** — 1-2 речення
- **Які варіанти розглядали** — список з 2-4 пунктів
- **Які труднощі / вузькі місця / слабкі сторони ідеї побачили** — що робить питання нетривіальним
- **Що вже відкинули і чому** — якщо є
- **Де сумніваємось / зайшли у глухий кут** — саме тому питаємо Gemini

Мета: Gemini не починає з нуля, а підключається до обговорення на тому етапі де ми зупинились.

Це резюме формує САМ Claude з пам'яті чату — не треба читати файли чи логи, бо обговорення тільки що відбувалось у цьому чаті.

## Крок 4 — Вивести промпт одним код-блоком у чаті

Вивести ОДНИМ код-блоком з чотирма backticks (щоб triple-backticks всередині не ламали рендер):

````markdown

Відповідай **українською**, стисло, по суті. Якщо знаходиш проблему — вкажи файл і рядок. Якщо все ок — скажи чітко "готово" або "згоден з підходом".
````

## Крок 5 — Інструкція Роману

Після виводу код-блоку написати Роману коротко:

> **Готово.** Тапни кнопку Copy над код-блоком → відкрий Gemini (застосунок на iPhone або https://aistudio.google.com) → вибери модель **Gemini 3 Pro** → встав. Коли отримаєш відповідь — встав її сюди, я проаналізую.

## Важливо

- **НЕ** виконувати реальних змін у коді цим скілом. Скіл тільки збирає контекст і формує промпт.
- **НЕ** викликати MCP-інструменти (`ask_gemini`, `gemini_review_code`). Цей підхід ручний.
- **НЕ** включати в промпт `node_modules/`, `bundle.js`, `.git/`, бінарники.
- **НЕ** створювати HTML-файл (правило скасовано у сесії nudNp 24.04.2026 — довгі списки/промпти виводяться тільки як код-блок у чаті).
- **НЕ** передавати `РОМАН_ПРОФІЛЬ.md` у промпт (Роман у сесії nudNp 24.04 сказав що це лишнє для Gemini).
- **НЕ** передавати `NEVERMIND_BUGS.md` у промпт (Роман у сесії nudNp сказав — баги не релевантні для ідей/концепцій).
- Якщо git diff занадто великий (>100K токенів) — попередити Романа і запропонувати звузити (тільки останній коміт замість всіх від main).
- Якщо Роман не задоволений відповіддю Gemini — обговорити з ним чому, не просто погоджуватись.
