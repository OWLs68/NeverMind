# /gemini — Запит другої думки від Gemini (ручна вставка в aistudio.google.com)

**Аргумент:** питання до Gemini (необов'язково). Приклади:
- `/gemini перевір мої зміни перед деплоєм`
- `/gemini я не знаю як краще зробити OWL reaction, запропонуй архітектуру`
- `/gemini` (без аргументу — тоді спитати Романа що він хоче від Gemini)

## Як це працює (концепція)

Це **не автоматичний MCP**. Це ручний workflow:
1. Claude збирає повний контекст проекту + git diff + питання Романа
2. Створює HTML-файл `gemini-<тема>.html` у корені з кнопкою «📋 Копіювати весь промпт» (правило CLAUDE.md «Довгі списки/шпаргалки/промпти — ЗАВЖДИ через HTML з кнопкою»)
3. Комітить + пушить → через ~3 хв файл на проді (`owls68.github.io/NeverMind/gemini-<тема>.html`)
4. Роман відкриває URL на iPhone → тапає кнопку → відкриває https://aistudio.google.com → вставляє → читає відповідь Gemini
5. Роман повертає відповідь Gemini в чат → Claude реагує (погоджується або аргументує)

Інфраструктура нульова. Ключі ніде не засвічуються. Gemini 3 Pro має 1M+ контекст — вміщає весь NeverMind.

**Виняток:** якщо Роман САМ каже «закинь у чат», «не треба html», «без файлу» — вивести промпт одним markdown-блоком у чаті (формат з чотирма backticks). Це узгоджено з правилом CLAUDE.md «Виняток: якщо Роман САМ каже "покажи у чаті" — пишемо в чаті».

## Крок 1 — Отримати питання від Романа

Якщо після `/gemini` є текст — це питання, іди до кроку 2.

Якщо аргумент порожній — запитати:
> **Що ти хочеш від Gemini?** (наприклад: "перевір перед деплоєм", "запропонуй архітектуру для X", "знайди баги в моєму підході до Y")

Чекати відповідь Романа перед переходом до кроку 2.

## Крок 2 — Зібрати контекст

Виконати по порядку:

1. **Прочитати `CLAUDE.md`** (правила проекту — повний файл)
2. **Прочитати `_ai-tools/SESSION_STATE.md`** (поточний стан сесії)
3. **`git branch --show-current`** — дізнатись поточну гілку
4. **`git diff main...HEAD`** — зміни від main до HEAD поточної гілки
   - Якщо diff порожній — пропустити, не включати в промпт
5. **`git diff main...HEAD --name-only`** — список змінених файлів
   - Для кожного файлу зі списку — **прочитати повністю** (Read tool). Винятки:
     - `bundle.js` — НЕ читати (згенерований, не потрібен Gemini)
     - `package-lock.json` — НЕ читати (занадто великий, не потрібен)
     - Бінарні файли (зображення, шрифти) — НЕ читати

## Крок 4 — Інструкція Роману

Після виводу код-блоку написати Роману коротко:

> **Готово.** Скопіюй блок вище (кнопка Copy у правому куті) і встав у Gemini (застосунок на iPhone з вибраною моделлю **Pro**, або https://aistudio.google.com з вибраною моделлю **Gemini 3 Pro**). Коли отримаєш відповідь — встав її сюди, я проаналізую і скажу що думаю.

## Важливо

- **НЕ** виконувати реальних змін у коді цим скілом. Скіл тільки збирає контекст і формує промпт.
- **НЕ** викликати MCP-інструменти (`ask_gemini`, `gemini_review_code`). Цей підхід ручний, через веб-інтерфейс Gemini.
- **НЕ** включати в промпт `node_modules/`, `bundle.js`, `.git/`, бінарники.
- Якщо gitу diff виявиться занадто великим (>100K токенів) — попередити Романа і запропонувати звузити (наприклад тільки останній коміт замість всіх від main).
- Якщо Роман не задоволений відповіддю Gemini — обговорити з ним чому, не просто погоджуватись.
