# /deploy — Деплой (публікація на сервер)

Виконай всі кроки по порядку. Не пропускати жодного.

## Крок 1 — Час
Запусти `date` в терміналі. Запам'ятай результат — він потрібен для CACHE_NAME.

## Крок 2 — CACHE_NAME (назва кешу)
Відкрий `sw.js` і зміни `CACHE_NAME` на формат `nm-YYYYMMDD-HHMM` (рік-місяць-день-година-хвилина).
Приклад: `nm-20260403-1831`.
НЕ вигадувати час — використовувати результат з кроку 1.

## Крок 3 — STATIC_ASSETS (список файлів для кешу)
Перевір що всі нові файли додані в масив `STATIC_ASSETS` у `sw.js`.
Якщо в цій сесії створювались нові файли — вони мають бути там.

## Крок 4 — Перевірка синтаксису
Запусти `node --check` для кожного зміненого `.js` файлу.
Якщо є помилка — СТОП. Спочатку виправити, тільки тоді далі.

## Крок 5 — Оновлення документації (після кожної сесії)

### Завжди — `_ai-tools/SESSION_STATE.md`
- **Оновлено:** сьогоднішня дата
- **Зараз робимо:** що було зроблено в цій сесії
- **Останні сесії:** додати рядок про сьогодні (коротко: що зроблено)
- **Відкриті баги:** оновити якщо щось закрито або з'явилось нове

### Якщо додавались нові файли або змінювались правила — `CLAUDE.md`
- Таблиця "Файлова структура" — додати новий файл з описом
- Розділ "ВСІ ПРАВИЛА" — якщо змінювались або додавались правила
- Таблиця "Дані (localStorage)" — якщо додавались нові ключі

### Якщо змінювалась архітектура (структура програми) — `docs/ARCHITECTURE.md`
- Оновити діаграми залежностей між модулями
- Оновити опис потоку даних якщо змінився

### Якщо змінювались відомі баги — `_ai-tools/NEVERMIND_BUGS.md`
- Закрити баг якщо виправлено
- Додати новий баг якщо знайдено
- Оновити статус якщо частково виправлено

## Крок 6 — Git push (відправка змін)
```
git add .
git commit -m "опис що зроблено"
git push -u origin <назва-поточної-гілки>
```

## Крок 7 — Перевірка деплою
Якщо CI (система автоматичного деплою) не спрацював через ~3 хвилини:
```
git commit --allow-empty -m "ci: retrigger" && git push origin <назва-поточної-гілки>
```
