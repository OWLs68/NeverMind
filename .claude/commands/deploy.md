# /deploy — Деплой (публікація на сервер)

Виконай всі кроки по порядку. Не пропускати жодного.

## Крок 1 — Час
Запусти `date` в терміналі. Запам'ятай результат — він потрібен для CACHE_NAME (назви кешу).

## Крок 2 — CACHE_NAME (назва кешу — версія збережених файлів)
Відкрий `sw.js` (Service Worker — файл що керує кешем браузера) і зміни `CACHE_NAME` на формат `nm-YYYYMMDD-HHMM` (рік-місяць-день-година-хвилина).
Приклад: `nm-20260403-1831`.
НЕ вигадувати час — використовувати результат з кроку 1.

## Крок 3 — STATIC_ASSETS (список файлів для кешування)
Перевір що всі нові файли додані в масив `STATIC_ASSETS` у `sw.js`.
Якщо в цій сесії (сеансі роботи) створювались нові файли — вони мають бути там.

## Крок 4 — Перевірка синтаксису (правильності коду)
Запусти `node --check` для кожного зміненого `.js` файлу.
Якщо є помилка — СТОП. Спочатку виправити, тільки тоді далі.

## Крок 5 — Оновлення документації

### ЗАВЖДИ — `_ai-tools/SESSION_STATE.md` (стан поточної сесії)
- **Оновлено:** сьогоднішня дата
- **Зараз робимо:** що було зроблено в цій сесії
- **Останні сесії:** додати рядок про сьогодні (коротко що зроблено)
- **Відкриті баги:** оновити якщо щось закрито або з'явилось нове

### Якщо змінювались або знайдені нові баги — `NEVERMIND_BUGS.md`
- Закрити баг (помилку) якщо виправлено — перемістити в розділ "Закриті"
- Додати новий баг якщо знайдено — в потрібну категорію (🔴 критичний / 🟡 середній)
- Оновити статус якщо частково виправлено

### Якщо змінювались правила або файлова структура — `CLAUDE.md`
- Таблиця "Файлова структура" — додати новий файл з описом
- Розділ "ВСІ ПРАВИЛА" — якщо змінювались або додавались правила
- Таблиця "Дані (localStorage — локальне сховище браузера)" — якщо додавались нові ключі

### Якщо змінювалась архітектура (структура програми) — `NEVERMIND_ARCH.md`
- Оновити діаграми (схеми) залежностей між модулями (частинами програми)
- Оновити опис потоку даних якщо змінився

### Якщо змінювалась логіка конкретної вкладки (Finance, Evening, Health...) — `CONCEPTS_ACTIVE.md`
- Оновити опис концепції (ідеї реалізації) тієї вкладки яка змінювалась

### Якщо з'явились нові заплановані фічі (функції) або скасовані плани — `FEATURES_ROADMAP.md` (дорожня карта функцій)
- Оновити статус фічі: запланована / в роботі / скасована

### Якщо дізнались щось нове про вподобання або стиль роботи Романа — `РОМАН_ПРОФІЛЬ.md`
- Додати нову перевагу або уточнити існуючу

### Якщо додались нові контекстні файли або змінилась структура проекту — `START_HERE.md` (точка входу нового чату)
- Оновити таблицю ситуаційних файлів якщо щось додалось або перемістилось

## Крок 6 — Git push (відправка змін на сервер)
```
git add .
git commit -m "опис що зроблено"
git push -u origin <назва-поточної-гілки>
```

## Крок 7 — Перевірка CI (системи автоматичного деплою)
Якщо CI не спрацював через ~3 хвилини:
```
git commit --allow-empty -m "ci: retrigger" && git push origin <назва-поточної-гілки>
```
