# /prompt-engineer — Єдиний формат для 10+ промптів OWL

Цей скіл активується коли треба змінити/тюнити промпт OWL у будь-якому місці коду. Стандартизує структуру промпту і ловить галюцинації до того як вони доїдуть у продакшен.

---

## ⚠️ Перед виконанням — обов'язкові перевірки

1. **НЕ міняти промпт "на відчуття" без тесту.** AI-промпти — чутлива зона. Зміна формулювання на 5 слів може зламати всі `tool_calls`. Перед правкою — спочатку прочитай промпт цілком + перечитай `getOWLPersonality()` (бо вона додається до більшості промптів і вже містить "ЗАЛІЗНЕ ПРАВИЛО", "ПРАВИЛО ЕМПАТІЇ", "ПРАВИЛО ЧЕСНОСТІ", "ПРАВИЛО G12").

2. **Не дублюй правила що вже є в персоналії.** `getOWLPersonality()` вже задає:
   - Перспективу (ти — OWL, юзер — "ти")
   - Заборону на матюки і образи
   - Антигалюцинацію (не вигадуй поточний стан з історії)
   - Ліміт мікро-розмов (2-3 повторення)
   
   Якщо додаєш нове правило про те саме — значить існуюче не спрацьовує і треба чинити його, не дублювати у новому промпті.

3. **Карта всіх промптів** (актуально на 17.04.2026):
   | Файл | Рядок | Промпт | Що робить |
   |------|-------|--------|-----------|

   | `src/ai/prompts.js` | 81 | `INBOX_SYSTEM_PROMPT` | Inbox classifier (task vs event vs project vs moment…) з tool calling |
   | `src/ai/prompts.js` | 156 | `INBOX_TOOLS` | 31 tool definitions — **джерело правди → `docs/AI_TOOLS.md`** |
   | `src/ai/prompts.js` | 213 | `getOwlChatSystemPrompt()` | OWL міні-чат (з CHIP_PROMPT_RULES) |

   Перед зміною — знайти ВСІ промпти в таблиці які зачіпає задача. Часто треба правити 2-3, не один.

4. **Анти-галюцинація у контексті.** Якщо додаєш новий блок у `getAIContext()` (`src/ai/core.js`):

   > **Tools:** перед додаванням або зміною tool definition — ЗАВЖДИ перечитай `docs/AI_TOOLS.md` (довідник 39 існуючих + 14 у планах). Оновлюй таблицю у цьому файлі після змін.
   - Додай явне правило у самому блоці: "НЕ питай це — вже знаєш" або "Це ІСТОРИЧНІ дані, не цитуй поточний стан".

## Стандартна структура промпту (формат Єдиний Шаблон)

Кожен OWL-промпт має ці блоки у такому порядку:

```
[1. РОЛЬ]        — хто ти, що робиш (1-2 речення)
[2. КОНТЕКСТ]    — що AI отримує ({getOWLPersonality} + getAIContext + специфіка)
[3. ПРАВИЛА]     — що ОБОВ'ЯЗКОВО робити (пріоритет, порядок перевірок)
[4. АНТИГАЛЮЦИНАЦІЇ] — що ЗАБОРОНЕНО (не вигадуй факти, не перепитуй задане)
[5. ПРИКЛАДИ]    — 2-4 короткі демонстрації (input → tool/text)
[6. ФОРМАТ ВИХОДУ] — що повертати (tool_calls, 1-3 речення текстом, конкретні поля)
```

### Коли який блок обов'язковий

| Тип промпту | Роль | Контекст | Правила | Анти-гал | Приклади | Формат |
|-------------|------|----------|---------|----------|----------|--------|
| Класифікатор (INBOX) | ✅ | ✅ | ✅ ПРІОРИТЕТ | ✅ | ✅ 5+ | ✅ tool call |
| Табло OWL | ✅ | ✅ | ✅ | ✅ ЧЕСНІСТЬ | — | ✅ JSON |
| Task/Project/Me chat | ✅ | ✅ | короткі | — | — | текст N речень |
| Аналіз (day/summary) | ✅ | ✅ | структура | ✅ | — | ✅ формат |
| Міграція даних | ✅ | — | детальні | — | ✅ 2-3 | ✅ tool call |

## Крок 2 — Відповісти письмово на 3 питання ПЕРЕД правкою

1. **Що саме зараз ламається/не так?** Конкретний приклад input → неправильний output.
2. **Який блок Єдиного Шаблона за це відповідає?** (Роль / Контекст / Правила / Антигал / Приклади / Формат)
3. **Чи проблема не у `getOWLPersonality()` або `getAIContext()` замість конкретного промпту?** Якщо так — чинити там, не тут.

## Крок 3 — Показати Роману diff ДО правки


## Крок 4 — Тест після зміни (мінімум 3 input-кейси)

1. **Позитивний кейс** — той що ламався → тепер працює.
2. **Негативний кейс** — той що НЕ має спрацьовувати → лишається некерований.
3. **Регресійний кейс** — сусідня функціональність що НЕ мала постраждати → не змінилась.

Якщо нема можливості тестувати локально — написати Роману список кейсів для ручної перевірки.

## Чек-ліст якості промпту (пройти перед пушем)

- [ ] Роль OWL явна (1 речення хто ти і що робиш)
- [ ] Правила мають пронумеровану ПРІОРИТЕТНІСТЬ якщо їх 3+
- [ ] Анти-галюцинація явна (хоча б 1 пункт "НЕ вигадуй X якщо немає Y")
- [ ] Приклади вказують і input і очікуваний output (не тільки один)
- [ ] Формат виходу однозначний (tool_call VS текст — без сірої зони)
- [ ] Немає дублювання правил з `getOWLPersonality()`
- [ ] Немає `undefined` / порожніх плейсхолдерів у шаблоні (перевірка `.includes('undefined')` у callAI)

## Антипатерни — чого НЕ робити

- ❌ **"Будь корисним і ввічливим"** — порожнє, нічого не дає. Конкретизуй ("2-3 речення, без вступів").
- ❌ **"Відповідай природньо"** — природньо = довго. Задавай число речень.
- ❌ **"Якщо щось не зрозуміло — запитай"** — ламає антигалюцинацію. Краще "якщо X відсутнє — зроби Y".
- ❌ **Емодзі в промпті без контексту** — модель може почати ляпати їх скрізь.
- ❌ **"Думай крок за кроком"** в робочому промпті — збільшує latency без користі для коротких відповідей.

## Важливо

- **НЕ** додавати нові tool definitions у `INBOX_TOOLS` (`src/ai/prompts.js`) без обговорення з Романом. Зараз 31 tool — ще +5 і модель починає плутатись. Перед додаванням — перечитай `docs/AI_TOOLS.md`.
- **НЕ** міняти `getOWLPersonality()` заради одного флоу — це ядро всіх 10+ промптів.
- Якщо треба нова перспектива агента — додавай новий persona у `personas` (поряд з coach/partner/mentor), не рефактор.
- У `src/ai/core.js` функції `callAI`, `callAIWithHistory`, `callAIWithTools` — не чіпати підпис без перевірки всіх 10+ місць виклику.
