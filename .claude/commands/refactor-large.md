# /refactor-large — Рефакторинг великих файлів

**Коли використовувати:** файл у проекті став завеликим (>500 рядків для JS, >1000 для HTML/CSS), його важко орієнтуватись, зміни часто ламають сусідні блоки. Роман каже "розбити файл" / "занадто великий" / "винести у модулі".

**Без аргументів або з назвою файлу:** `/refactor-large` або `/refactor-large src/tabs/finance.js`.

---

## ⚠️ Перед виконанням — обов'язкові перевірки

1. **Чи стабільний поточний код?** Не рефакторити якщо є відкриті 🔴/🟡 баги у `NEVERMIND_BUGS.md`. Спочатку закрити, потім рефакторити. Рефакторинг поверх битого коду = дабл-баги.
2. **Чи немає свіжих нетривіальних змін у файлі?** Подивитись `git log --oneline -10 <file>`. Якщо за останні 3 дні багато змін — Роман ще може щось міняти, зачекати.
3. **Чи є план?** Створити `_ai-tools/REFACTORING_<MODULE>.md` з розбивкою на файли + порядок фаз + оцінка ризиків. **Показати Роману ДО початку** — отримати "Роби".
4. **Чи впевнений що backward compat збережеться?** Всі зовнішні модулі мають працювати БЕЗ змін імпортів. Стратегія — re-exports з оригінального файлу.
5. **CACHE_NAME bump обов'язковий у кінці.**

---

## Основний робочий процес

### Крок 1 — Карта файлу

Читати весь файл частинами (по 400 рядків) і скласти карту:
- Список усіх `export function` / `export const`
- Список внутрішніх функцій (приватних)
- Список `let`/`const` (state)
- Window-exports (inline `onclick` у HTML)
- Зовнішні імпорти (хто з інших модулів імпортує звідси)

**Приклад з finance.js (2443 рядки, винесено у 6 модулів):**
```
Секція 1 (34-121): SVG-іконки + палітра → finance-cats.js
Секція 2 (123-327): CRUD категорій → finance-cats.js
...
```

### Крок 2 — План фаз

Запропонувати Роману план. Кожна фаза:
- **Одна логічна область** (наприклад "Аналітика") → один новий файл
- Порядок: **від найменш ризикової до найбільшої** (чиста логіка → UI → модалки з shared state)
- Backward compat: оригінальний файл **re-exports** усе перенесене

**Шаблон плану:**
```
| Фаза | Файл | Рядки | Ризик |
|------|------|-------|-------|
| 1 | module-a.js | ~280 | 🟢 чиста логіка |
| 2 | module-b.js | ~400 | 🟢 ізольована |
| 5 | module-modals.js | ~1100 | 🟡 shared state |
```

### Крок 3 — Стратегія "skeleton + Edit" (КРИТИЧНО)

**Чому:** один великий `Write` >250 рядків у Claude Code Web часто обриває **stream idle timeout**. `Edit` блоки по 50-100 рядків майже завжди проходять.

**Дія для кожного нового файлу:**

1. **Write скелет** (~15-30 рядків):
   ```js
   // imports
   // state
   // заглушки: export function foo() {}
   // window-exports
   ```

2. **Edit** замінює кожну заглушку реалізацією (50-100 рядків кожна):
   ```
   old_string: export function foo() {}
   new_string: export function foo() { /* реальна логіка */ }
   ```

3. **Build перевірка:** `node build.js` після кожної фази. Помилки → фіксити одразу.

### Крок 4 — Checkpoint-коміти після кожної фази

**КРИТИЧНО:** не чекати кінця всіх фаз.

Після кожної фази:
```bash
git add <змінені-файли>
git commit -m "refactor(<module>): phase N — <name>"
# push не обов'язковий кожну фазу, але перед перервою — обов'язковий
```

**Чому:** якщо посеред Фази N обірветься stream — Фази 1..N-1 вже у git. Новий чат продовжить з Фази N без втрати прогресу.

### Крок 5 — Фінальний аудит

Після всіх фаз:
1. `node build.js` — чистий, без помилок
2. `wc -l` — перевірити що оригінальний файл зменшився до цільового розміру
3. `grep -rn "from ['\"].*finance" src/` — перевірити всі зовнішні імпорти резолвляться
4. **CACHE_NAME bump** у `sw.js`
5. **Оновити документацію:**
   - `CLAUDE.md` → "Файлова структура" (нові файли)
   - `_ai-tools/SESSION_STATE.md` → підсумок рефакторингу
   - `docs/CHANGES.md` → запис з датою
6. Фінальний коміт + push
7. Повідомити Роману **обов'язковий ручний smoke-test у браузері** — перелік функцій які треба протестити

## Анти-патерни (що НЕ робити)

- ❌ **Один великий Write >300 рядків коду** — обірве таймаут. Skeleton + Edit.
- ❌ **Коміт тільки в кінці всіх фаз** — ризик втрати роботи при обриві посеред.
- ❌ **Змінювати імпорти у зовнішніх модулях** без re-exports у оригіналі — ламає backward compat, багато файлів треба правити.
- ❌ **Обіцяти Роману "потестити у браузері" — але не казати що саме** — він не знає все перевіряти. Давати **конкретний список 7-10 пунктів** з критичними сценаріями.
- ❌ **Рефакторити "для краси"** без реальної проблеми (Роман сказав "занадто великий"). Не вигадувати задачу, тільки на прохання.
- ❌ **Рефакторити + додавати нові фічі одночасно** — змішує ризики. Спочатку рефакторинг → коміт → окремо нова фіча.
