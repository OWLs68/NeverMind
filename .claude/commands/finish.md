# /finish — Оновлення документації перед новим чатом

> **Скоуп:** тільки проект **NeverMind**. Скіл живе у `.claude/commands/finish.md` (проектний рівень), не в `~/.claude/commands/` (глобальний). В інших репо його немає — навмисно.

**Коли використовувати:** коли сесія логічно завершена і треба підготувати репо до нового чата. Можна викликати **кілька разів** в одному чаті — другий виклик дописує все нове що обговорилось після першого.

**Без аргументів.** Просто `/finish`.

---

## Філософія скіла

`/finish` НЕ підсумок для Obsidian (це `/obsidian`). `/finish` — це **оновлення репозиторію** так щоб наступний чат через `/start` отримав повний контекст:

- Актуальний стан у `_ai-tools/SESSION_STATE.md`
- Свіжі закриті баги у `NEVERMIND_BUGS.md`
- Оновлений `ROADMAP.md` (Active/Done)
- Запис у `docs/CHANGES.md`
- Нові концепції/ідеї які обговорились — у правильні файли

**Ідемпотентність:** скіл перевіряє чи блок поточної сесії вже існує. Якщо нема — перший виклик, створює блок + робить ротацію. Якщо є — повторний виклик, тільки оновлює існуючий блок свіжими даними (нові коміти, нові обговорення), ротацію не повторює.

## Фаза 0 — Архівація SESSION_STATE (ПЕРШИМ кроком, до підсумку)

> **Чому першим:** архівація = швидка детермінована операція (Read + Edit на 2 файли). Підсумок Фаз 2-9 = довга аналітична робота. При обриві потоку (stream timeout — обрив відповіді на 90%+ контексту) краще втратити підсумок (відновлюється з транскрипту повторним викликом `/finish`) ніж архівацію (накопичується = борг у живому SESSION_STATE.md, push блокується Hook №1 з SK6E2 при >2 активних блоках). Введено SK6E2 29.04.2026 після 4 сесій підряд (kGX6g→UG1Fr→m4Q1o→oknnM) де архівація летіла на високому контексті 3 рази з 4.

**Тільки у режимі CREATE.** В UPDATE — пропустити Фазу 0, одразу до Фази 1.

1. **`git branch --show-current`** → ідентифікатор поточної сесії (`XXXXX` з `claude/start-session-XXXXX`). Потрібен щоб НЕ винести випадково власний блок (його ще нема, але страховка).
2. **Прочитати `_ai-tools/SESSION_STATE.md`** — знайти всі заголовки `^## 🔧 (Поточна сесія|Сесія|Попередня сесія) XXXXX` що НЕ містять «архівовано».
3. **Якщо активних блоків ≤ 2** — пропустити архівацію (правило ще не порушено). Перейти до Фази 1.
4. **Якщо активних блоків > 2** — взяти найстарший (за датою у заголовку) і винести у `_archive/SESSION_STATE_archive.md`:
   - Якщо архів не існує — створити з заголовком + дата створення + посилання назад на `SESSION_STATE.md`
   - Додати блок сесії **зверху** архіву (хронологічно останні вгорі)
5. У живому `SESSION_STATE.md` **видалити** перенесений блок і **замінити** на stub-рядок: `## 🔧 Сесія XXXXX (DD.MM.YYYY) — архівовано {назва_поточної} {DD.MM} → [archive](../_archive/SESSION_STATE_archive.md#anchor)`.
6. Якщо після виносу все ще >2 активних — повторити кроки 4-5 для наступного найстаршого, поки не лишиться рівно 2.

**→ Чекпоінт-коміт:** `docs(session-state): archive {ID_винесених} (phase 0 of /finish {XXXXX})`. **ОБОВ'ЯЗКОВО до Фази 1.** Push можна тимчасово відкласти до Фази 7, але коміт локально — зараз.

## Фаза 1 — Збір контексту сесії

1. **`git branch --show-current`** → ідентифікатор сесії (`XXXXX` з `claude/start-session-XXXXX`). Якщо гілка не в форматі `claude/start-session-*` — взяти останній суфікс після `-`.
2. **`git log <гілка> --not main --oneline`** → коміти сесії (які ще не в main)
3. **`git log --all --oneline --since="today 00:00"`** → все сьогодні (контроль)
4. **`git diff main --stat`** → список змінених файлів
5. **`cat deploy-counter.txt`** + **бейдж у `index.html`** → діапазон версій сесії
6. **Пошук `CACHE_NAME` у `sw.js`** → поточна мітка
7. **Переглянути розмову сесії цілком — ОДНИМ проходом** і витягти **ВСІ** факти у внутрішню чернетку (memory scratchpad — тимчасова пам'ять у голові Claude, не файл) перед форматуванням у секції. Категорії що збираються одночасно:
   - **Зроблено у коді** (фічі/фікси/рефакторинги — для Фази 2.A «Зроблено» і Фази 9 «Що зроблено»)
   - **Обговорено без виконання** (концепти/ідеї/сумніви/плани — для Фази 2.A «Обговорено» і Фази 9 «Що обговорили»)
   - **Ключові рішення** («вирішили X бо Y», «відмовились від Z» — для Фази 2.A «Ключові рішення» і Фази 9)
   - **Рішення Романа** (особисті цитати, що відхилив, що обрав з варіантів — для Фази 9 «Рішення Романа»)
   - **Інциденти** (reset/force push/збої CI/відкати з hash — для Фази 2.A «Інциденти» і Фази 9)
   - **Конфлікти/суперечності** (де план змінився посеред сесії, де Роман передумав — для Фази 9)
   - **Відкладене** (свідомо відклали + орієнтир + причина — для Фази 9)
   - **Спостереження стилю Романа** (швидкість/настрій/фокус — для Фази 9)
   - **Відкриті питання** (не вирішено у цій сесії)

**КРИТИЧНО:** НЕ робити окремий аналіз історії per-секцію (per-секцію — для кожної секції окремо). ОДИН прохід → внутрішня чернетка з 9 категоріями → потім форматування у Фази 2.A і 9. Це найбільша економія часу /finish (введено 29.04.2026, заміняє раніше неявний девʼятикратний прохід).

**Перевірка ідемпотентності:** шукати у `_ai-tools/SESSION_STATE.md` блок з ID поточної сесії. Якщо є → **режим UPDATE** (Фаза 2.B). Якщо нема → **режим CREATE** (Фаза 2.A).

## Фаза 2.A — `SESSION_STATE.md` CREATE (перший виклик)

> **Архівація вже зроблена у Фазі 0.** Тут тільки створення блоку поточної сесії і оновлення секцій.

1. **Додати блок поточної сесії зверху** за шаблоном (з внутрішньої чернетки Фази 1):
   ```markdown
   ## 🔧 Сесія XXXXX — {короткий заголовок теми} ({DD.MM.YYYY})
   
   ### Зроблено
   - [список 3-10 пунктів по темах]
   
   ### Обговорено (без виконання)
   - [концепти, ідеї, плани на майбутнє]
   - [якщо нема — "Без додаткових обговорень."]
   
   ### Ключові рішення
   - [формат "вирішили X, бо Y"]
   - [якщо нема — "Без окремих рішень."]
   
   ### Інциденти (ОБОВ'ЯЗКОВО — мінімум one-liner)
   - [reset/force push/збої/відкати з hash — конкретно]
   - [якщо нема — рівно "Без інцидентів." (3 слова, не розгорнутий ритуал)]
   
   ### Метрики
   - Коміти: {перший_hash} → {останній_hash} ({N} комітів)
   - Версії: v{від} → v{до}
   - CACHE_NAME: {поточне}
   - Build: {чистий/з помилками}
   ```

   **Правило D (one-liner для порожніх):** замість «Без `git reset` / `git push --force` / без skip hooks. Усі коміти першою спробою.» — рівно «Без інцидентів.» Якщо є — описати конкретно з hash. Сигнал «секція є і порожня» збережено, ритуальні фрази усунено.

2. **Оновити секцію "⚠️ ДЛЯ НОВОГО ЧАТУ — найважливіше"** — 5 пунктів найкритичнішого для наступного чата (замінити старі).
3. **Оновити "🎯 НАСТУПНИЙ КРОК"** — на основі поточного стану ROADMAP і відкритих задач.
4. **Оновити таблицю "Проект"** (версія, hash, CACHE_NAME, гілка).
5. **Додати рядок у "📦 Попередні сесії"** — короткий опис поточної сесії.

## Фаза 6 — Умовні оновлення (тільки якщо зміни релевантні)

Скіл перевіряє чи були у сесії такі зміни. Якщо так — **пропонує Роману конкретні правки** у відповідні файли і чекає підтвердження ПЕРЕД записом:

| Умова у сесії | Файл |
|---------------|------|
| Новий JS-файл у `src/`, новий `localStorage` ключ, нова "недоторканна річ" | `CLAUDE.md` (секції файлової структури / даних / "Що не можна змінювати") |
| Концепція вкладки змінила % (70→100 і т.п.) | `CONCEPTS_ACTIVE.md` |
| Нові потоки даних / event types / міжмодульні залежності | `docs/ARCHITECTURE.md` |
| Нова модалка / палітра / компонент | `docs/DESIGN_SYSTEM.md` |
| Додано/видалено/перейменовано скіл | `_ai-tools/SKILLS_PLAN.md` + `START_HERE.md` |
| Змінилась файлова структура верхнього рівня | `START_HERE.md` |
| **Закрита фаза з планового документу** (`_archive/OWL_SILENCE_PRUNING_PLAN.md`, `docs/EVENING_2.0_PLAN.md`, `docs/FINANCE_V2_PLAN.md`, `docs/AGENT_INTELLIGENCE_SCALE.md`, інші `docs/*_PLAN.md`) | відповідний `docs/*_PLAN.md` — позначити фазу `✅ ВИКОНАНО ({сесія} {дата})` з посиланнями на коміти |

**Правило:** для `CLAUDE.md` — ЗАВЖДИ питати дозвіл. Для інших файлів — можна правити автоматично, у звіті згадати що саме змінено.

**→ Коміт:** окремий для кожного файла з префіксом `docs({назва}): sync session {XXXXX}`.

## Фаза 7 — Коміт і пуш

1. `git status` — перевірити що в staging area
2. **Не додавати:** `bundle.js`, код у `src/`, `index.html`/`style.css`/`sw.js` (це робота сесії, не handoff)
3. Коміти після кожної фази вже зроблені (Фази 2-6)
4. Якщо щось залишилось незакоміченим — окремий коміт `docs: finish session {XXXXX} cleanup`
5. **Push з retry:** `git push -u origin <гілка>`. При network error — exponential backoff (2s, 4s, 8s, 16s), до 4 спроб
6. НЕ створювати pull request автоматично

## Фаза 9 — Obsidian-блок для RoamBrain (ОБОВʼЯЗКОВО після Фази 8)

> **Обʼєднано з `/obsidian` 24.04.2026 сесія v2vYo за прямим запитом Романа.** Тепер `/finish` одним викликом і оновлює репо, і дає готовий блок для копіювання у RoamBrain. Раніше Роман чекав `/finish` + `/obsidian` окремо — це було довго.

Одразу після звіту Фази 8 **ВИВЕСТИ окремим повідомленням** — код-блок у форматі нижче (щоб у чаті була кнопка Copy). Роман копіює і вставляє у чат з Claude у репозиторії `roman-brain` (там Claude сам класифікує і покладе у `projects/nevermind/nm-YYYY-MM-DD.md`).

**Філософія (з ex-`/obsidian` скіла):** RoamBrain — це архів для майбутніх розслідувань. Через місяць-рік треба буде зрозуміти "коли це почалось?" / "хто зробив X?" / "чому так?". Тому підсумок **ширший за "що Роман явно обговорював"**: усі технічні інциденти, усі обговорення навіть без рішень, усі дивні ситуації. Приклад коли це рятує — 14.04.2026 між двома сесіями зникли деплої v54-v130 через reset без запису у daily notes, розслідування 17.04 не знайшло автора.

**Формат блоку:**

```

Нема секції взагалі — невідомо: чи не сталось, чи Claude забув через обрив на 95% контексту. - Інциденти **НЕ приховувати** — якщо Claude зробив щось незаплановане, ЦЕ МАЄ бути у підсумку. Чесність важливіша за гарний вигляд.

## Правила виконання

1. **Ідемпотентність — КРИТИЧНО.** Перевірка блоку поточної сесії у SESSION_STATE = єдине джерело правди для режиму CREATE/UPDATE.
2. **Архів — тільки при CREATE.** UPDATE ніколи не чіпає архіви.
3. **CLAUDE.md — завжди з підтвердженням Романа.** Пропонувати правку як патч, чекати "так/ні".
4. **Checkpoint-коміти між фазами.** Правило №2 з CLAUDE.md "🔧 Робочий процес" — при обриві stream timeout не втрачаємо прогрес.
5. **Мова звіту** — українська. Технічні терміни — у дужках людською.
6. **Фаза 9 обовʼязкова.** Obsidian-блок генерується ЗАВЖДИ. Раніше був окремий `/obsidian` — тепер обʼєднано у `/finish` (24.04.2026 v2vYo).
7. **НЕ** чіпати `bundle.js`, код у `src/`, `index.html`/`style.css`/`sw.js` — це робота сесії, а не handoff.
8. **Архівація — Фаза 0, ПЕРЕД підсумком.** При обриві потоку на >90% контексту: підсумок (Фази 2-9) відновлюється повторним викликом `/finish` з транскрипту. Архівація (Фаза 0) — НЕ відновлюється сама, накопичується як борг у живому SESSION_STATE.md, push блокується Hook №1 при >2 активних блоках. Корінь проблеми зафіксовано SK6E2 29.04.2026 після 4 сесій підряд (kGX6g→UG1Fr→m4Q1o→oknnM) де архівація летіла на високому контексті 3 рази з 4.
9. **Один прохід по транскрипту (Фаза 1 пункт 7).** ОДНА читка → внутрішня чернетка з 9 категоріями фактів → потім форматування у Фази 2.A і 9. НЕ робити окремий аналіз історії per-секцію (це найбільша економія часу /finish).

## Що робити якщо щось не так

- **Конфлікт merge при push** → НЕ робити reset/force. `git pull --rebase origin <гілка>`, вирішити руками, ще раз push. Якщо конфлікт у `SESSION_STATE.md` — значить інший чат теж робив `/finish` паралельно, зливати обидва блоки сесій.
- **Сесія без комітів** (тільки обговорення) → все одно виконати Фази 2-5 (SESSION_STATE з "Код не чіпали", CHANGES з обговореннями). Архіви ротуються.
- **Два `/finish` підряд без змін** → другий просто скаже "нічого нового не додано, архів не чіпався".
- **ID сесії не з гілки** (наприклад гілка `main`) → використати `session-{YYYYMMDD-HHMM}` як ID.
