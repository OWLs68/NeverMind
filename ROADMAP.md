- **Динамічний розпорядок дня** (додано 10.04) — **ключовий принцип:** розпорядок = реальна картина дня на конкретну дату, а НЕ абстрактний шаблон тижня. Зараз "кожен понеділок о 7:00 підйом" (шаблонний) — треба динамічний прив'язаний до дат. Як має працювати:

  2. **Детектор конфліктів** — новий блок накладається на існуючий → OWL перепитує: "У тебе на 15:00 уже зал. Заміняємо на зустріч чи переносимо зал?"

### Блок 5 — Мозок OWL: якість рішень

- **4.4 Relevance Scoring (локальний)** (автор: GPT-4o, складність: мала 1 сесія) — бали доречності 0-10 без API: повернувся після 30 хв = +3, дедлайн сьогодні = +3, чат відкритий = −3, ігнорував 3 табло поспіль = −5. Якщо бал < 5 → НЕ генерувати табло. Прибирає "набридливу муху". Доповнює існуючий Judge Layer новим шаром локальних балів.
- **4.20 + 4.39 Negative Memory + Override** (консолідовано: автори GPT-4o + Grok розширені, складність: мала-середня 1 сесія) — зберігати не лише "що спрацювало" а й "що дратує / ігнорується". Новий ключ `nm_negative_rules`:
  - Структура: `{trigger: "task:презентація", context: "вівторок 9:00", action: "never_remind"}`
  - **Авто-додавання правил** коли юзер ігнорує тему 3+ разів
  - **Кнопка UI "Не нагадуй мені про це"** на повідомленнях OWL (один тап)
  - Передача правил у промпт AI → Fast Path не генерує такі пропозиції
  - Антиспам для OWL. Агент розумнішає з кожним днем
- **4.6 Contextual Personality Shifts** (автор: GPT-4o, складність: мала 1 сесія) — OWL адаптує не тільки тон а й ПОВЕДІНКУ. `getContextMode()` повертає один з:
  - `planning` (зранку) = OWL пропонує план
  - `flow` (юзер у потоці, багато дій за короткий час) = OWL мовчить
  - `struggling` (після невдачі, багато переносів) = підтримка
  - `celebrating` (після перемоги, закрив складне) = святкує
  - `reflecting` (вечір) = підсумки
  - `returning` (повернувся після паузи) = welcome-back
- **G1 Тиша як модифікатор** (автор: Gemini, уточнення Judge Layer) — тиша 60 хв НЕ самодостатній тригер. Підсилює інші сигнали, але без контенту (нема задач/звичок/дедлайнів) = 0 балів. OWL мовчить якщо нема про що — навіть після 3 годин тиші. Фікс Judge Layer.
- **G2 Burst Detection** (автор: Gemini, антиспам серій) — юзер закрив 3+ задач за 2 хв → OWL мовчить, збирає дії у `burstQueue`. Через 3 хв тиші — одне резюме "🔥 5 задач за 20 хвилин" з чіпами. Алгоритм:
  - `onUserAction()` → додає в чергу
  - Блокує Judge Layer для `data-changed`
  - Показує instant reaction (локально)
  - `clearTimeout + setTimeout(3m)` → `processBurstQueue()` → якщо 4+ дій = один API-запит з масивом виконаного (замість 3-5 окремих)
- **G3 Характери → різний поріг Judge Layer** (автор: Gemini, різна ПОВЕДІНКА а не тільки слова):
  - **Тренер:** поріг 4 (рідше говорить, різко), дедлайн = +10, похвала за дрібниці = 0
  - **Партнер:** поріг 2 (частіше на зв'язку), тиша = більше балів, нагадує про перерви
  - **Ментор:** поріг 3, активний на зміні фаз (ранок/вечір), фокус на рефлексії
- **G10 Velocity tracking (темп юзера)** (автор: Gemini) — JS рахує скільки задач юзер зазвичай закриває до обіду (середнє за тиждень). Якщо сьогодні значно менше — Judge Layer отримує тригер `velocity-drop (+3)`. "Повільний день. Потрібна допомога?" [Розбити задачу] [Показати найлегшу].
- **4.21 Verify Loop (перевірка після дії)** (автор: GPT-4o, складність: мала) — після дії коротко перевіряти результат а не просто "зроблено". Наприклад: створив задачу → перевірив що вона в списку → підтвердив юзеру. Робить агента менш тупим.

- **4.34 Memory Echo (ехо пам'яті)** (до Supabase) — раз на 4-6 годин OWL тихо згадує щось особисте з минулого в контексті сьогодні: "Ти казав що ненавидиш пізні зустрічі. Сьогодні дзвінок о 9:30 — перенести?". Створює відчуття що OWL справді ПАМ'ЯТАЄ тебе. Реалізується через `nm_facts` + просте правило в промпті. Майже безкоштовно.
- **4.35 Anticipatory Micro-Actions (випереджаючі мікро-дії)** (до Supabase) — OWL сам робить маленькі підготовчі дії: створює шаблон задачі, додає нагадування, позначає "можливо важливо" — і просто питає підтвердження одним чіпом. Юзер відчуває що агент працює на випередження.

#### Ідеї DeepSeek (07.04.2026)

- **4.44 Event Sourcing (журнал подій)** (патерн для Supabase, статус: на розгляді) — вся взаємодія = потік незмінних подій. AI бачить тільки НОВІ події яких не бачив. Менше токенів, менше галюцинацій. Як Redux для AI.
- **4.45 CSS/SVG стартова анімація** (до Supabase, статус: прийнято) — базові рухи сови (дихання, моргання, поворот голови) через CSS transition + SVG морфінг. Безкоштовно, легко, як **стартовий варіант до Lottie/дизайнера**.
- **4.46 Ембеддинг-антиповтор** (після Supabase, статус: на розгляді) — порівнювати вектори значення тексту замість слів. Ловить перефразування ("як день?" ≈ "чим займаєшся?"). Потребує Transformers.js або API.

#### Ідеї Gemini для пізніх фаз (09.04.2026)

- **G7 Focus Mode (режим фокусу)** (після Supabase) — "Start" біля задачі → PWA переходить у Zen Mode. Все зникає, залишається лише задача + таймер + OWL. Judge Layer блокує ВСЕ крім "Закінчив" / "Здаюся". Інструмент для глибокої роботи.
- **G8 speechSynthesis для критичних** (після Supabase) — OWL озвучує голосом дедлайни та нагадування коли Judge Layer score > 8. Не все — лише критичне. Web Speech API `speechSynthesis`, вбудовано у браузер. Миттєвий ефект Jarvis.
- **G14 Інтеграції через Supabase** (після Supabase, окреме обговорення) — Webhooks + Edge Functions: Google Calendar, Telegram, банківські API, Google Maps тощо. **⚠️ НЕ РОБИТИ без детального обговорення з Романом** — багато можливостей, треба пріоритизувати. OAuth для "Увійти через Google" → імпорт календаря.

## 🚫 Rejected — відхилені з причиною

**Бізнес-рішення (11.04):**
- **BYOK (Bring Your Own Key)** — користувач сам реєструється у OpenAI і вставляє ключ. Суперечить UX "мінімальне тертя", зайвий крок в онбордингу. Замість — freemium + API на сервері Supabase.
- **Assistants API (OpenAI)** — суперечить принципу 4.22 Micro-orchestration (багато дешевих рішень на 4o-mini замість одного важкого виклику). Обговорено з Gemini 11.04.

**Архітектурні рішення:**
- **Централізована БД-абстракція** (єдиний модуль `db.js` що обгортає весь localStorage) — вже пробували у старій флат-структурі (до рефакторингу), **зламало все одразу**. Кожна вкладка працює з localStorage напряму через власні функції `get*()` і `save*()`.

**Поведінка OWL:**
- **Поведінкова телеметрія** (трекінг часу на вкладках, ігнорування порад) — перебір. Ризик дратувати ("бачу, ти уникаєш фінансів"). Потребує дуже тонкого налаштування промптів.
- **Doubt Score** (лічильник сумнівів — створив і одразу видалив) — занадто нішево. Рідкісний кейс не варте окремої системи.

**Відхилено після обговорень з DeepSeek:**
- **Лінгвістичні правила "робо-стиль"** ("Молоко. Нагадати?") — занадто сухо. OWL має характер (Partner/Coach/Mentor), не телеграф.
- **Видалити табло** — табло = ініціатива OWL, core фіча NeverMind.
- **Видалити чіпи** — мінімальне тертя = core принцип NeverMind.

**Anti-patterns (помилки яких уникати, затверджено 09.04):**
- ❌ **"As an AI..."** — заборонити AI вибачатись і згадувати що він ШІ. Промпт: "НІКОЛИ не згадуй що ти ШІ"
- ❌ **Пасивна агресія** — "ти знову не виконав" ЗАБОРОНЕНО. Замість: "може дропнемо або розіб'ємо?"
- ❌ **Goldfish memory** — завжди передавати стан останніх 2-3 хвилин у промпт
- ❌ **API заради привітання** — мікро-реакції локально (масив фраз в JS), API тільки для смислових повідомлень
- ❌ **Набридливість** — OWL рідко говорить але завжди влучив (підтверджено дослідженнями NN/Group)

4. Edge Functions (серверні функції)
5. Push (сповіщення)
6. Storage (файли, аудіо)

**Відкладено на окреме обговорення ДО Supabase:**
- **SyncEngine архітектура** — як саме реалізувати offline-first sync (Варіант C за Gemini: localStorage синхронний, у фоні поштар відправляє на сервер). Окрема сесія проектування.
- **Деталізація freemium меж** — що саме у безкоштовній версії (скільки запитів AI на день? які вкладки доступні? базовий OWL без проактивності?), що у підписці. Впливає на UI онбордингу, Edge Functions rate limiting, систему обліку використання.
- **BroadcastChannel cleanup** — зняти `localStorage.setItem` override після перевірки що BroadcastChannel покриває всі кейси. Низький пріоритет — працює як є.

**Фічі що потребують серверу (від різних авторів):**

- **4.9 Pattern Detection** (оригінальний план) — щотижневий аналіз: "Найпродуктивніший у вівторок 10-12", "Коли спиш <7 год — наступного дня −40% задач", "Витрати на каву +40%". Edge Function аналізує `nm_habit_log2`, `nm_tasks`, `nm_health_log`, `nm_finance`, `nm_moments`. Результат: `nm_patterns`. Потребує серверних cron jobs.
- **4.10 Jarvis Insights (щотижневі картки)** (оригінальний план) — красива карточка з аналізом тижня: задачі, звички, кращий день, витрати, інсайт. Генерується в неділю. `nm_weekly_insight`. Спеціальний дизайн (не просто текст на табло). Потребує Pattern Detection.
- **4.11 Фонова рефлексія** (оригінальний план) — Edge Functions: коли юзер не в додатку, Jarvis аналізує дані, готує ранковий бриф, оновлює Day State, шукає патерни. Зранку відкриваєш — все готово.
- **4.12 Push-сповіщення з контекстом** (оригінальний план) — Jarvis стукає навіть коли додаток закритий: "Через годину суд, документи готові?". VAPID-ключі, SW push handler.
- **4.13 Emotional Intelligence по поведінці** (оригінальний план) — аналіз не слів а ДІЙ: зайшов-погортав-вийшов = демотивація, 5 задач за годину = потік (не перебивати), короткі різкі повідомлення = роздратований. `nm_behavior_signals`. Потребує аналітики на сервері.
- **4.14 Predictive Actions** (оригінальний план) — OWL пропонує ДО того як попросять: понеділок зранку → "Плануємо тиждень?", день до дедлайну → "Залишилось 3 кроки, зробити зараз?". На основі `nm_patterns` + календар.
- **4.15 Implicit Feedback Loop** (оригінальний план) — ігнор табло 3 рази → `save_memory("Не любить питання під час роботи")`. Ніколи не натискає чіпи з емодзі → перестати їх генерувати. Повне неявне навчання.
- **4.16 Трирівнева пам'ять (повна)** (оригінальний план) — Working Memory (10 реплік) → Day State (саммарі дня) → Core Memory (JSON по категоріях: [Робота], [Здоров'я], [Стиль]). Серверне оновлення Core Memory через Edge Functions. **Частково закривається 4.2** (структуровані факти вже є).
- **4.17 API-ключ на сервері** (оригінальний план) — юзер не вводить ключ вручну. Ключ на Supabase, запити через серверну функцію. Безпечніше. **Обов'язкове рішення** після Supabase (бізнес-модель freemium).
- **4.18 Єдина пам'ять між пристроями** (оригінальний план) — один Jarvis — телефон, комп'ютер, планшет. Той самий `nm_facts`, ті самі задачі, та сама історія.
- **4.23 Intent Graph (карта цілей)** (автор: GPT-4o) — зв'язати задачі, звички, події, фінанси в одну карту НАМІРІВ. OWL бачить не записи а цілі: "Хоче бігати" → звичка + задача "купити кросівки" + подія "забіг 5К". Один намір = кілька записів. Потребує БД для зв'язків.
- **4.24 Planner / Executor / Reviewer** (автор: GPT-4o) — розділити ролі AI: Planner вирішує що робити, Executor виконує, Reviewer перевіряє. Зараз змішано → шум. З Edge Functions можна робити послідовно.
- **4.26 Belief Graph (граф переконань)** (автор: GPT-4o) — кожен факт у пам'яті має `source` (звідки), `confidence` (0-1 впевненість), `age` (вік), `expiresAt` (термін придатності), `lastConfirmed` (коли підтверджено). OWL знає НАСКІЛЬКИ вірить у факт. "Роман працює в IT" (0.9) vs "Любить каву" (0.5, давно не підтверджувалось).
- **4.32 Voice-first (Realtime API)** (автор: GPT-4o) — OpenAI Realtime API: WebRTC/WebSocket, voice-to-voice без STT/TTS ланцюжка, VAD автоматично ловить початок/кінець мовлення. OWL звучить як розумна присутність.
- **4.37 IndexedDB замість localStorage** (автор: Grok) — localStorage має ліміт 5-10MB. IndexedDB практично безлімітний і швидший для великих даних. Міграція коли дані виростуть.
- **4.38 Offline-first sync** (автор: Grok) — при Supabase: local-first залишається священним. Supabase тільки для sync + auth + push. Конфлікт-резолвінг: **last-write-wins + user version vector** (простіше ніж CRDT).
- **4.42 Ghost Presence (примарна присутність)** (автор: Grok розширені) — SW + Push: OWL перевіряє дані і надсилає push з одним чіпом. "Ти зараз у flow-зоні. Закриваємо презентацію?" Відчуття що сова завжди поряд. **НА iOS ТІЛЬКИ ЧЕРЕЗ СЕРВЕР.**
- **4.43 Conversation Threading Фаза B** (автор: Grok розширені) — повний holistic аналіз з Pattern Detection (Фаза A локальна — у Блоці 6).
- **4.47 RAG через `search_facts(query)` tool** (додано 11.04, автор: Gemini обговорення) — замість запихати всі факти у `getAIContext()` — дати AI інструмент `search_facts(query)`. AI сам шукає потрібні факти коли їх треба. Розвантажує контекст і дозволяє масштаб 100+ фактів без token-budget хвороб. Після pgvector: embeddings → семантичний пошук → виклик `search_facts` з тексту юзера → повертає топ-5 релевантних. **Заміна тривіальна:** замінити `formatFactsForContext()` дамп на tool definition. Тех-боргу нуль — зараз `formatFactsForContext` простий варіант, коректний для поточного масштабу.
- **i18n — дві мови** — англійська + ще одна. `src/core/i18n.js` з `t('key')`, `nm_settings.lang`, перемикач у Settings UI. Зачіпає: UI тексти в `index.html` (~200-300 місць), JS тексти (~100), AI промпти (~25-30 двома мовами), OWL personality. Оцінка: 3-4 сесії.
- **🎨 AI-теми за промптом юзера** (запозичено з `theme-factory`, складність: середня 1-2 сесії) — юзер вводить опис ("хочу теплу осінню", "мінімал сіро-блакитний") → AI генерує цілісну палітру (primary, secondary, accent, bg, card-bg, text, text-muted, border, danger, success) через новий tool `generate_theme(description)`. Обов'язково: перевірка контрасту WCAG (щоб текст був читабельним), заборона фіолету у промпті, превью перед застосуванням. **Чому після Supabase:** сервер кешує популярні теми (не палимо API на кожен клік юзера), Edge Function перевіряє доступність, складніші моделі (4o) для якіснішої палітри. **Фіча-диференціатор** — Notion/Todoist такого не мають. Варіант монетизації платної підписки.
- **🦉 Інтерактивний туторіал з OWL-провідником** (додано 15.04.2026 сесія jMR6m, автор: Роман) — покрокове проходження базових функцій при першому запуску або після повного скидання даних. Алгоритм: підсвічується конкретний елемент UI (кнопка, поле вводу, картка), поряд стоїть OWL (анімований персонаж) і через speech-бабл пояснює що це і для чого. Юзер **реально виконує дію** (не просто читає — тапає поле, пише задачу, бачить створену картку). Після дії — наступний елемент підсвічується. Проводить по основних сценаріях: створення задачі, додавання звички, запис витрати, навігація між вкладками, взаємодія з OWL табло. **Чому саме після Supabase:** усі фічі мають бути готові а UI більше не мінятимемо — інакше туторіал застаріє одразу. **Що вже є:** кнопка "Як користуватися NeverMind" у налаштуваннях + `openHelp(tab)` по кнопці "?" у header кожної вкладки + `src/tabs/onboarding.js` (слайди при першому запуску) — все це модифікувати і доробити до повноцінного walkthrough. **Переваги над відео:** працює offline (PWA вимога), оновлюється разом з кодом (не застаріває), контекстний (показує реальний UI юзера), нуль зовнішніх залежностей. Оцінка: 2-3 сесії.

## 🧭 Архітектурні принципи

> Принципи що керують рішеннями — до яких звертатись коли вибираєш між варіантами реалізації.

**Від Grok (07.04.2026):**
- **"OWL рідко говорить але завжди влучив"** — Jarvis геніальний саме тому що мовчить. Якщо OWL спам — юзер перестане читати взагалі.
- **"OWL трохи менш розумний але набагато уважніший"** — якщо OWL здається розумнішим за юзера, юзер перестає довіряти. OWL = уважний друг, не всезнаюча система.
- **Ніяких графів, belief systems, multi-agent ролей (для MVP)** — Event Log + `nm_facts` + `nm_negative_rules` = достатньо. Переускладнення → maintenance hell для PWA.
- **AI Layer два виклики:** 4o-mini (3500 токенів, 95% рішень) + 4o/Claude (коли confidence < 65%).
- **Supabase Realtime** — використовувати не для всього, а тільки для критичних подій.

**Від Gemini (09.04.2026, обговорення стратегії):**
- **SyncEngine (Варіант C / 4.38 Offline-first) — шлях міграції на Supabase.** localStorage залишається синхронним джерелом істини UI, у фоні поштар відправляє на сервер через чергу. Читання — cache-first + тихе оновлення з сервера.
- **Велосипед vs стандарт:** зараз велосипед виправданий (PWA без бекенду), після Supabase — перехід на pgvector/RAG (стандарт).
- **Zep/Mem0 naming** — використовувати структуровані факти з timestamps а не наративне саммарі (зробили у 4.2).

**Від NeverMind CLAUDE.md (критичні):**
- **Один мозок на все** — OWL = єдиний Jarvis. Табло, чати, чіпи — різні "вікна" одного мозку. Зміна в одній вкладці = зміна в усіх.
- **Мінімальне тертя** — користувач пише будь-що в Inbox, ШІ автоматично класифікує. Без ручного вибору типу.
- **Single source of truth** — кожна вкладка читає з свого ключа localStorage напряму. Без дублювання даних між ключами.
- **Без "Роби" від Романа — код не чіпати.** Правило №1.

| Structured Outputs: перший запит з новою схемою ПОВІЛЬНІШИЙ (парсинг схеми). Повторні — нормальні | github.com/vercel/ai | Тримати схеми стабільними і перевикористовувати |
| "AI silence" дослідження: мовчазна згода НЕ підвищує довіру. Краще сказати "Згоден" одним рядком ніж мовчати | lancashire.ac.uk | Instant reactions ("Зроблено! 💪") = правильний підхід |
| GitHub Copilot: "ghost text" — найменш нав'язливий формат. Ніколи не відкриває повний UI | docs.github.com | Можливо: chips як ghost-елементи що зникають якщо ігноруються |
| Arc browser: AI в sidebar, не в popup. Manual invocation + lightweight suggestions | seraphicsecurity.com | Табло = sidebar-підхід (не popup). Правильно |
| ChatGPT memory: юзер бачить що AI знає, може видалити. "Silent memory" + periodic "I noticed..." | aicompetence.org | Memory Pane — UI де юзер бачить `nm_facts` і редагує (вже є з 11.04) |
| Дослідження переривань: AI може перебивати ТІЛЬКИ коли: (1) контекст-релевантно (2) вартість пропуску висока (3) AI визнає поточну задачу юзера | essv.de | 3 умови перед кожним табло. Вшити в Judge Layer |
| NN/Group: промпти/підказки працюють коли РІДКІ і ГІПЕР-РЕЛЕВАНТНІ. Надмірність → юзер ігнорує ВСЕ | nngroup.com | "OWL рідко говорить але завжди влучив" = підтверджено |
| Ambient Intelligence: працює тихо у фоні, дає маленькі пасивні affordances замість повних промптів | aiuxdesign.guide | Наші instant reactions = правильний ambient підхід |

**Ключовий висновок:** наші рішення (Judge Layer, Relevance Scoring, Override Memory, Tool Calling, Memory Pane, rare-but-relevant OWL) підтверджуються реальним досвідом провалів Clippy/Google Now і успіхів ChatGPT/Arc/Copilot.
