**Правило:**
1. **Перед оцінкою складності або планом** — СПОЧАТКУ прочитай реальний код всіх залучених файлів. Ніколи не давай оцінку "на пам'ять" або з поверхневого скану.
2. **Перед першою відповіддю на нову задачу** — прочитай файли які стосуються задачі (код, не тільки документацію). Це включає: imports (імпорти — що файл підтягує з інших модулів), залежності, суміжні модулі.
3. **Краще одна якісна відповідь ніж три поверхневі.** Якщо не впевнений — читай більше, а не відповідай швидше. Роман витрачає час на виправлення кожної поверхневої відповіді.

## 🔧 Робочий процес (надійність проти обривів)

> **Додано 17.04.2026 gHCOh** після рефакторингу finance.js. Протягом сесії обриви stream idle timeout обрізали відповіді 5+ разів, що ледве не привело до втрати прогресу. Ці правила — щоб наступна велика задача **пережила обриви без втрати роботи**.

### Правило №1 — Великі файли через skeleton+Edit

**Тригер:** створюєш JS/HTML-файл де попередній `Write` буде >250 рядків.

**Дія:**
1. `Write` скелет з заглушками — імпорти + `export function foo() {}` плейсхолдери + window-exports (~15-30 рядків)
2. Для кожного логічного блоку — окремий `Edit` який замінює заглушку реалізацією
3. Після всіх Edit — `build` перевірка
4. При обриві посеред — продовжую з останньої заглушки, попередні блоки не постраждали

**Чому:** один великий `Write` 400+ рядків у Claude Code Web часто обриває stream idle timeout. `Edit` блоки по 50-100 рядків майже завжди проходять бо коротші.

**Виняток:** файли <250 рядків — одразу `Write`, без надобудов.

### Правило №2 — Checkpoint-коміти після кожної фази

**Тригер:** багатофазна робота (рефакторинг, міграція, великий фікс у кілька підсистем).

**Дія:** після кожної **завершеної логічної підзадачі** — коміт + push. Не чекати кінця всіх фаз.

**Приклад з рефакторингу finance.js (gHCOh):** 6 фаз по винесенню модулів. Якби після Фази 1 одразу коміт → Фази 2-4 → коміт → тоді посередині Фази 5 обрив не втратив би Фази 1-4 (вони вже на GitHub).

**Не плутати з правилом про SESSION_STATE.md:** те правило — про оновлення статусу документації. Це — про збереження коду.

### Правило №3 — Мінімум тексту між tool calls

**Тригер:** послідовність tool calls під час виконання задачі.

**Дія:** між tool calls писати **одне речення-максимум** (<25 слів). Описові абзаци економлять увагу юзера і зменшують ризик stream idle timeout **на текстовій частині** (обриви бувають і до першого tool call).

**Виняток:** перший і останній меседж у задачі — тут можна ширше (пояснення плану на старті + підсумок у кінці).

**Якщо не впевнений чи тригер спрацював** — краще викликати скіл ніж проігнорувати. Ціна помилки: пара зайвих секунд читання. Ціна ігнору: баг який скіл би запобіг.

## 🗺️ Робота з ROADMAP (пріоритизація нових ідей)

> **Проблема яку це вирішує:** коли Роман кидає нову ідею/концепцію/фічу, вона через "свіжість у контексті" (те що щойно обговорили — у голові Claude сильніше ніж старі пункти плану) автоматично потрапляє на верхівку `ROADMAP.md`. Потім Роман мусить переробляти порядок — рефакторинг плану (переробка структури) замість реальної роботи. Правило додано 15.04.2026 у сесії jMR6m за прямим запитом Романа.

**Правило:** перед тим як записати нову концепцію/фічу/план у `ROADMAP.md`:

1. **Прочитати поточний стан плану** — секції `🚀 Active`, перші 2-3 блоки `📋 Next`, відкриті баги у `NEVERMIND_BUGS.md` (щоб розуміти з чим порівнювати).

2. **Проаналізувати залежності** нової ідеї:
   - **Блокує?** — чи вона є передумовою (prerequisite — те що має бути зроблено до) для вже запланованого? Якщо так → ПЕРЕД тим що вона блокує.
   - **Залежить?** — чи вона потребує щось з поточного плану? Якщо так → ПІСЛЯ того від чого залежить.
   - **Незалежна?** → аналізуємо важливість окремо.

3. **Порівняти важливість** з сусідами у блоці. Чесно запитати: "Чи це важливіше ніж пункти X, Y, Z які вже стоять там?" Якщо ні — кінець відповідного блоку або `💡 Ideas`.

4. **Запропонувати Роману конкретне місце з обґрунтуванням** (НЕ просто "куди покласти?"):
   - _"Ця ідея логічно йде у Блок 2 після Здоров'я — незалежна від Active, не блокує нічого критичного."_
   - _"Ця ідея важливіша за пункти A, B у Блоці 1 — пропоную перемістити їх нижче. Чому: вона блокує цілу Фазу 2 Здоров'я."_

5. **Чекати підтвердження ДО запису** в `ROADMAP.md`. Не писати "на випередження" — інакше порушується принцип правила.

6. **Якщо Роман каже "додай першим"** — перепитати: _"Переконаний? Це відсуне пункти A і B. Чому вона важливіша?"_ Не для суперечки, а щоб Роман свідомо приймав рішення (не автоматично через "свіже в голові").

**Як це відрізняється від поточної поведінки:**
- **Було:** ідея → одразу запис у ROADMAP → пізніше рефакторинг порядку
- **Треба:** ідея → аналіз + пропозиція місця з обґрунтуванням → підтвердження Романа → запис

## Система деплою

**Коротко:** Claude пушить у `claude/**` → `auto-merge.yml` мержить у `main`, збирає bundle, деплоїть на GitHub Pages (2-3 хв). **Деталі (один job, `-X theirs`, concurrency, cache-bust правила):** → [`docs/TECHNICAL_REFERENCE.md`](docs/TECHNICAL_REFERENCE.md#система-деплою).

## AI-логіка

**Коротко:**
- **Inbox** використовує OpenAI Tool Calling (**47 tools:** 31 у `src/ai/prompts.js` + 8 нових UI у `src/ai/ui-tools.js` + 8 health/memory/cat. Повний довідник → `docs/AI_TOOLS.md`). AI повертає `msg.tool_calls[]`, диспатчиться через існуючі handlers (`processSaveAction`, `processFinanceAction`, `processCompleteHabit/Task`, `handleUITool`, `clarify`, `restore_deleted` тощо).
- **Tab chat bars і proactive.js** — ще на текстовому JSON форматі (`callAI()`, `callAIWithHistory()`, `callOwlChat()`).
- **`getAIContext()`** повертає: дата/час, профіль, `nm_facts` (структурована пам'ять), задачі/звички/inbox, фінанси, кошик, поточне повідомлення OWL табло.
- **OWL Board:** Табло (ініціатива агента, read-only) + Чат-бар (ініціатива юзера). Кожні 3 хв, 7-23, cooldown через `nm_owl_cooldowns`, пам'ять 30 повідомлень.

**Повний Inbox flow діаграма + список полів getAIContext + концепція OWL Board з усіма технічними деталями:** → [`docs/TECHNICAL_REFERENCE.md`](docs/TECHNICAL_REFERENCE.md#ai-логіка).
