# ⚖️ COUNCIL — Концепція (чернетка для нової сесії)

> **Статус:** не реалізовано. Це нотатка-результат обговорення Романа + Claude + Gemini у сесії `claude/start-session-rKQPT` (02.05.2026). Не створювати скіл / не міняти архітектуру до нової сесії з `/prompt-engineer`.

---

## 0. Звідки це взялось

Gemini розібрав Ромі відео про «раду радників» для AI-агентів. Ідея: Claude часто робить «здається ок» рішення які виявляються поверхневими через 2 сесії. Рада симулює внутрішній конфлікт експертів щоб ловити це до коміту або одразу після.

Ми пройшли **3 ітерації з Gemini**. Перші дві — Gemini обходив фундаментальні питання (чи це театр? чек-лист простіший? термін життя?). На третю — закрив 4 з 4 критичних дірок які Claude знайшов після ітерації v1.0.

---

## 1. Філософія (3 речення)

**Council** — інструмент примусового когнітивного тертя для Claude, що запобігає поверхневим рішенням («shortcut bias»). Він імітує внутрішній конфлікт ролей для виявлення «сліпих плям» у межах однієї моделі. Council вмирає і мігрує у простий чек-лист коли Рома стабільно не отримує нових інсайтів (👍 < 60% за 2 тижні).

**Чим відрізняється від `/gemini`:**
- Council — внутрішній голос Claude (локально, миттєво, без API costs)
- `/gemini` — зовнішній консультант (інша модель, для глухих кутів)

**Чому це частково театр (чесно):**
LLM має одну модель світу. 5 ролей фізично не можуть мати 5 різних онтологій — це 5 обгорток одної думки. Користь з'являється ТІЛЬКИ якщо правила суворі (обов'язковий Якір, `#SkipReason`). Без правил — балаканина.

---

## 2. Термін життя (важливо!)

**Це тренувальна конструкція на 3-6 місяців.** Виправдана зараз, бо NeverMind — перший застосунок Роми, він не знає тонкощів. Стратег і Свіжий погляд підкидають варіанти яких чек-лист дати не може.

**Через рік** коли Рома знатиме проект — Council стане баластом. План виходу — у секції H (метрика смерті).

## 3. ФІНАЛЬНА СПЕКА v1.1 (Gemini)

### A. Філософія
Див. розділ 1 цього документа.

### B. Тригери

**Явний:**
- `/council [ідея/план]` — перед реалізацією
- `/council --review [код]` — після батча Claude
- `/council --quick` — швидка версія

**Auto-Warning (🚨)** — Claude зобов'язаний запропонувати `/council` якщо:
- змінюється `package.json` або додаються зовнішні CDN-скрипти
- **localStorage Schema:** змінюються ключі або структура об'єктів у `setItem`/`getItem` (детектиться порівнянням поточної логіки з існуючою)
- обсяг змін > 150 рядків за один батч
- змінюється логіка `sw.js` або `manifest.json`

Формат пропозиції: `🚨 Це one-way door, запустити /council?`

### C. Режими роботи

- **`/council` (Повний):** для архітектурних змін та нових фіч. **5 ролей.**
- **`/council --quick` (Швидкий):** для дрібних правок. **3 ролі: 🕵️ Критик + 🛠 Виконавець + ⚖️ Голова.**
  - (NB: Оптиміст у quick — баласт. План реалізації важливіший за оптимізм для дрібниць.)

### D. Ролі та обмеження

Кожна роль — **макс. 2 речення + обов'язковий Якір** (файл/рядок/змінна).

1. 🕵️ **Критик** — шукає дірки, ліміти localStorage, race conditions
2. 🌀 **Стратег** — пропонує радикально інший шлях (наприклад, CSS замість JS)
3. 🚀 **Оптиміст** — шукає можливості масштабування та майбутній профіт
4. 💎 **Свіжий погляд** — UX, iOS-специфіка, офлайн-стани
5. 🛠 **Виконавець** — конкретний технічний алгоритм (1-2-3)

> **Правило #SkipReason:** якщо роль не має що додати — пише `Пропускаю (#NoLogic / #NoUI / #Legacy)`. БЕЗ причини — пропуск не зараховується.

### E. Логіка Голови (Conditional Synthesis)

Голова **не зважує** думки, а формулює рішення через умови:

> *Критик правий ЯКЩО [умова_X], Оптиміст правий ЯКЩО [умова_Y]. У нашому випадку [X/Y], тому робимо [Z].*

Без формулювання «у нашому випадку Х» — синтез провалено → No Echo тригер (секція I).

### F. Формат виводу (mobile-first)

**Реальний приклад на задачі «Додати темну тему»:**

```
### ⚖️ Council: Додати темну тему

* 🕵️ Критик: Пряма маніпуляція класами в JS викличе Flash of Unstyled Content (FOUC).
  Якір: UI.js:init
* 🌀 Стратег: Використовуй суто CSS-змінні та color-scheme, без JS-слухачів.
  Якір: styles.css:root
* 🚀 Оптиміст: Це дозволить легко додати "High Contrast" режим пізніше.
  Якір: Theme.js
* 💎 Свіжий погляд: Треба врахувати системний перемикач iOS у matchMedia.
  Якір: manifest.json
* 🛠 Виконавець: 1) Визначити змінні в :root. 2) Додати data-theme на body.
  3) Зберегти вибір у localStorage.

Конфлікт: Критик (FOUC) vs Стратег (CSS-only).
⚖️ Голова: Критик правий щодо мерехтіння, Стратег правий щодо чистоти.
Рішення: кладемо скрипт вибору теми в <head> ДО рендеру body, використовуючи CSS-змінні.

Log: 2026-05-02 | DarkMode | Head-script + CSS-vars | styles.css

👍 чи 👎?
```

### G. Persistence (`_ai-tools/COUNCIL_LOG.md`)

- **Файл:** `_ai-tools/COUNCIL_LOG.md`
- **Пошук перед запуском:** Claude шукає **5 найрелевантніших** записів (за ключовими словами / файлами / задачею) — НЕ просто останніх 5. Захист від повторення помилок.
- **Формат запису:** `| Дата | Задача | Рішення | Якір | Vote |`

### H. Метрика та 👍/👎 флоу

1. Після виводу Council — Claude пише: **«👍 чи 👎?»** одним рядком
2. Рома відповідає одним символом у чаті
3. **Claude САМ** дописує цей символ у відповідний рядок `COUNCIL_LOG.md` (Рома НЕ редагує файл — він з телефону, це тертя)
4. **Трансформація:** якщо 👍 < 60% за 2 тижні → скорочуємо ролі до 2 (Критик + Виконавець)
5. **Критерій смерті:** якщо останні 5 записів мають 👎 → Claude пропонує видалити скіл і перенести досвід у `checklist.md`

### I. Anti-Patterns

- **No Fluff:** заборонені фрази «було б непогано», «врахуйте», «зверніть увагу». Тільки факти + Якір.
- **No Ghosting:** не починати кодинг, поки Голова не закрила сесію Council.
- **No Echo (🔄 Trigger):** Claude **зобов'язаний** написати `🔄 Глухий кут, рекомендую /gemini` ЯКЩО:
  - 🕵️ Критик написав `Пропускаю` АБО не знайшов жодного ризику
  - ⚖️ Голова не може сформулювати умову конфлікту (всі ролі згодні)

## 4. Що Claude (я) ще бачу як можливі дірки v1.1

Не критично для MVP, але варто перевірити перед коммітом скіла:

1. **Тригер «localStorage Schema»** — як саме Claude детектить зміну схеми? Чи достатньо grep по `setItem`/`getItem` у diff? Чи треба whitelist ключів? Уточнити у новій сесії.
2. **Лог >50 записів через 2 місяці** — пошук «5 найрелевантніших» через ключові слова може промахнутись. Чи робити теги вручну? Чи покладатись на семантичний пошук Claude?
3. **No Echo тригер** — якщо Критик завжди шукатиме хоч якийсь ризик щоб не активувати тригер → буде хибна тривога. Як цьому запобігти?
4. **Конкуренція з `/audit`, `/fix`** — у Роми вже є ці скіли. Чи перетинаються? Уточнити мapping коли який скіл вибирати.
