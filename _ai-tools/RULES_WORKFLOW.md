# RULES_WORKFLOW — процес роботи

> Читай при будь-якій задачі що змінює код/документи. Як читати, як планувати, як перевіряти результат.

## 1. Спочатку читай — потім говори

**Проблема яку це вирішує:** Claude систематично дає поверхневі відповіді «на пам'ять» — оцінює складність без читання коду, пропонує плани без розуміння залежностей. Роман витрачає 2-3 повідомлення на «розкачку» кожну сесію.

### Правила

1. **Перед оцінкою складності або планом** — СПОЧАТКУ прочитай реальний код всіх залучених файлів. Ніколи не давай оцінку «на пам'ять» або з поверхневого скану.
2. **Перед першою відповіддю на нову задачу** — прочитай файли які стосуються задачі (код, не тільки документацію). Включно з: imports (імпорти — що файл підтягує з інших модулів), залежності, суміжні модулі.
3. **Краще одна якісна відповідь ніж три поверхневі.** Якщо не впевнений — читай більше, а не відповідай швидше.
4. **Самоперевірка перед відповіддю:** «Чи читав я реальний код? Чи моя оцінка базується на фактах чи на припущеннях? Чи буде Роман задоволений глибиною цієї відповіді, чи скаже "копни глибше"?»

## 2. Чекліст повноти для архітектурних задач

**Тригер (хоча б одне):**
- Задача має план-документ (`docs/X_PLAN.md` — наприклад `OWL_SILENCE_PRUNING_PLAN.md`, `EVENING_2.0_PLAN.md`, `FINANCE_V2_PLAN.md`)
- Створюється нова підсистема / engine (engine — ядро функціональності) / layer (layer — архітектурний шар)
- Роман у запиті використовує: «переробка / великий план / багато фаз / масштабне / архітектура / новий модуль»
- Під час роботи виявилось що задача більша ніж здавалась (почалось як один баг, виявилось 5 модулів)

### Дія до «Роби»: 4-пунктовий чекліст вголос

1. **Усі call-sites** (місця виклику) залучених функцій — конкретний список з файлами та рядками. Не «приблизно 5», а «8 місць: `inbox.js:171`, `habits.js:1373`...».
2. **Сусідні модулі що зачіпаються побічно** — через спільне сховище (`localStorage`-ключі), події (`nm-data-changed` тощо), імпорти.
3. **Edge cases (граничні випадки)** які я **явно** перевірив — null / порожні масиви / дублі ID / гонки оновлень / lazy migration.
4. **Існуючий схожий патерн у проекті** який можна повторити (наприклад «робимо як `addEventDedup` у `calendar.js`») — щоб не вигадувати новий.

**Якщо хоч пункт не заповнений — стоп, повертайся читати.** Як саме читати (Explore-агент / прямий Read / Grep) — твій вибір під задачу. Чекліст про **результат**, не про інструмент.

**Чому додано (27.04.2026 C8uQD):** Explore-агент знайшов що в `getAIContext()` нема ID-маркерів типу `[task_xxx]` поряд з назвами сутностей — без них Pruning Engine не зміг би заповнити `entityRefs`. «Швидке дослідження» (grep по 2-3 файлах) це б проґавило, і Фаза 2 будувалась би у порожнечу.

## 3. UI smoke test після міграцій ID/формату/схеми + після нових AI tools що пишуть дані

**Тригер (хоча б одне):**
- Міграція торкається даних які рендеряться у DOM (`onclick`, `data-id`, селектори, `String(id)` порівняння)
- Зміна формату ідентифікаторів (`Date.now()` → UUID, числовий → рядковий, тощо)
- Зміна схеми localStorage що використовується у HTML-шаблонах
- Глобальна заміна типу поля у структурі даних
- **Нова AI tool яка пише у localStorage** (`create_task`, `create_event`, `add_transaction`, `save_habit`) — особливо якщо bulk (>1 запис за виклик: серії, повторення, batch-import). Ціна помилки множиться.

### Дія перед пушем — ВРУЧНУ перевірити основні UI-шляхи на iPhone

- Тап ✓ на елементі **рукою** (не через AI чат)
- Свайп вліво для видалення **рукою** (не через AI чат)
- Введення тексту у поле + збереження
- Тап на картку → відкриття модалки → закриття
- Drag-n-drop або інші специфічні взаємодії якщо є
- **Для нових AI-tools що генерують дані:** реальна репліка юзера → перевірити що AI створив **точно те що очікувалось**. Особлива увага до **неоднозначних запитів** («щотижня пн ср пт» без години — AI не повинен вигадувати 19:00).

**НЕ покладатися лише на AI-шлях через tool calls.** Модель може успішно викликати `complete_task` попри зламаний DOM. Юзер на iPhone тапає рукою — і нічого не працює.

### Підтверджені кейси

- **маскот сови v284** — махання сови не спрацювало на проді (тільки локально перевірено)
- **B-94/95** — промпт не перебив архітектуру, тестувалось через AI а не повний шлях
- **B-108** — UUID-міграція (xGe1H 27.04): `onclick="toggleTaskStatus(${id})"` з UUID-string давав ReferenceError. Сесія не перевірила тап ✓ рукою — пушнули, юзер 12 годин не міг закривати задачі ручно.
- **Календар Phase 2 рекурентність** (kGX6g 29.04): tool формально працював у dev. iPhone-тест: AI вигадав 19:00 для «щотижня пн ср пт» + створив 36 копій без warning про конфлікт. Відкат `2043a48`.

## 4. Edit/Delete/Cleanup у плані фічі

**Паттерн «менше > бардак» підтверджений тричі за квітень. Створення без cleanup = напівфабрикат → відкат.**

**Тригер:** будь-яка фіча де AI створює дані / структуру / нові записи (нова tool типу `create_*` / `add_*` / `repeat_*`, новий тип у localStorage, генерація серії або bulk-операція).

### Дія до «Роби»

План обов'язково включає **три речі** — створення, редагування, видалення (cleanup-механізм). Якщо плануєш тільки створення — **зупинись і додай решту**.

### Що рахується cleanup-механізмом

- Tool `delete_*` для відповідного типу (через AI чат)
- UI-кнопка / свайп-видалення в інтерфейсі (рукою)
- Для bulk — окреме видалення серії одним дотиком (не по одному запису)
- Migration / wipe для legacy-даних якщо схема міняється

### Підтверджені кейси

- **Маскот сови rSTLV 18-19.04** — додали анімацію без можливості виключити / прибрати. Видалено повністю.
- **`delete_event_series` tool kGX6g 29.04** — Роман відхилив на етапі обговорення бо «складність без миттєвої цінності».
- **Календар Phase 2 рекурентність kGX6g 29.04** — `repeat_weekly` без warning конфлікту і без cleanup для серії → 36 копій з вигаданим часом → повний відкат `2043a48`.

### Як виглядає у плані

Замість «додам tool `repeat_weekly`» → «додам tool `repeat_weekly` (створює повторну подію щотижня) + warning при конфлікті + tool/UI для видалення серії одним дотиком + захист від AI-вигадування години».

## 5. Перша спроба не спрацювала — СТОП

Перечитати код, зрозуміти ЧОМУ не спрацювало, тільки тоді наступна спроба. **Не пробувати варіації без розуміння причини.**

**Корінь vs симптом:** якщо 2-3 спроби лагодити одне і те ж не дають результату — зроби крок назад: «а проблема та?». Часто корінь у іншому місці. Класична помилка — патч на патч на симптом замість того щоб подивитися ширше.

**Урок з 01.05 (6ANWm):** ціла сесія витрачена на лагодження хука `check-response-violations` (whitelist, поріг блокування, anti-loop) замість того щоб запитати «а навіщо нам цей хук взагалі якщо CLAUDE.md занадто великий і я його все одно поверхнево читаю?». Корінь — у структурі документації, не у хуку.

## 6. Перевірка на внутрішню узгодженість після точкової правки

Коли Роман просить змінити конкретне місце (не глобальну переробку) — після виконання **швидко проглянь суміжний контент** (інші місця того самого файлу, пов'язані правила, інструкції поряд): чи не залишились згадки старого варіанту, чи правила/описи тепер не суперечать новому формулюванню.

### Два типи знахідок — різна реакція

1. **Те саме семантично** (наприклад правило «Останній рядок завжди: X» поряд з шаблоном який містить X) — виправляй одразу у тій самій правці. Інакше правило буде суперечити самому собі.
2. **Суміжне але поза межами запиту** (наприклад інструкція знизу яка описує старий флоу) — НЕ виправляй сам, а **ФЛАГУЙ Роману** з конкретним місцем, цитатою старого тексту і пропозицією нового. Він вирішує — оновлювати чи залишити.

**Причина:** поважає межі запиту (Роман попросив конкретну річ, не глобальний рефакторинг) + ловить «гнилі» залишки (rot — старі формулювання які накопичуються коли проект еволюціонує а документи не встигають).

## 7. Робота з ROADMAP (пріоритизація нових ідей)

**Проблема:** коли Роман кидає нову ідею/концепцію, вона через «свіжість у контексті» автоматично потрапляє на верхівку `ROADMAP.md`. Потім Роман мусить переробляти порядок.

### Правило (додано 15.04.2026 jMR6m)

Перед тим як записати нову концепцію/фічу/план у `ROADMAP.md`:

1. **Прочитати поточний стан плану** — секції `🚀 Active`, перші 2-3 блоки `📋 Next`, відкриті баги у `NEVERMIND_BUGS.md`.
2. **Проаналізувати залежності** нової ідеї:
   - Блокує? (передумова для вже запланованого) → ПЕРЕД тим що вона блокує
   - Залежить? (потребує щось з поточного плану) → ПІСЛЯ того від чого залежить
   - Незалежна? → аналізуємо важливість окремо
3. **Порівняти важливість** з сусідами у блоці. Чесно: «Чи це важливіше ніж пункти X, Y, Z?» Якщо ні — кінець відповідного блоку або `💡 Ideas`.
4. **Запропонувати Роману конкретне місце з обґрунтуванням** (НЕ просто «куди покласти?»):
   - _«Ця ідея логічно йде у Блок 2 після Здоров'я — незалежна від Active.»_
   - _«Ця ідея важливіша за пункти A, B у Блоці 1 — пропоную перемістити їх нижче. Чому: вона блокує цілу Фазу 2 Здоров'я.»_
5. **Чекати підтвердження ДО запису** в ROADMAP.md.
6. **Якщо Роман каже «додай першим»** — перепитати: «Переконаний? Це відсуне A і B. Чому вона важливіша?». Не для суперечки, а щоб свідомо приймав рішення.
