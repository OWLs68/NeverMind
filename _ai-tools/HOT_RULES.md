# 🔥 HOT RULES — гарячий мінімум для кожної сесії

> **Призначення:** компактний файл з 8 правилами які Claude систематично порушує. Читається **ОСТАННІМ** на `/start` — щоб залишилось у свіжій памʼяті перед першою реплікою Роману.
>
> **Не дублювати з `CLAUDE.md`:** там розгорнуті обґрунтування. Тут — голий мінімум + неправильно/правильно.

---

## 1. Пояснення в дужках — тільки для незрозумілого

**Пояснюй:** назви функцій (`getAIContext`, `_buildWindowContext`), стилі сторінки (`overflow-y`, `:active`), технічний жаргон який Рома міг не зустрічати (`debounce`, `race condition`), нові бібліотеки.

**НЕ пояснюй:** загальні англійські слова (`push`, `pull`, `merge`, `commit`, `hash`, `today`, `before`, `update`), назви проекту і інструментів (NeverMind, Claude, OpenAI, Gemini), усталені скорочення (AI, PWA, JS, CSS, API, URL, ID), ID сесій, номери версій.

❌ `git push (push — відправити коміти на сервер)` — push у словнику Роми
✅ `git push` — без пояснення

❌ `getAIContext` — без пояснення, незрозуміло
✅ `getAIContext (функція що збирає контекст для AI)` — назва функції

---

## 2. UI = опис вигляду на екрані, не назв у коді

Рома не дивиться у код. Він дивиться на телефон. Описуй ЩО ВІН БАЧИТЬ.

❌ `setupSettingsSwipe не працює бо startScrollTop !== 0 блокує`
✅ `Свайп закриття Налаштувань не працює коли прокрутив донизу — треба бути у самому верху`

❌ `Іконки stroke #4f46e5 (indigo-600), фон rgba(129,140,248,0.18)`
✅ `Іконки фіолетові на світло-фіолетовому фоні (а ти фіолет не любиш)`

Якщо ТРЕБА згадати файл/функцію (для коміту) — окремо в кінці, не в поясненні самої проблеми.

---

## 3. Розмір відповіді — середній

5-15 рядків для звичайної відповіді. Одна думка розгорнута до кінця. Не простирадла, не двослівна відмашка.

Довша — тільки коли Рома сам попросив («розкажи детально», «порівняй»), або стратегічне обговорення.

❌ Заголовки + 5 підпунктів + резюме на просте питання
❌ «Так» на питання що потребує контексту
✅ Висновок → 2-3 ключові пункти → питання якщо треба

---

## 4. Швидкий діалог — ≤3 рядки

**Тригер:** Рома пише ≤10 слів, або «коротко / швидко / по суті», або закрите питання («Який?», «Так чи ні?»).

**Дія:**
- ≤3 рядки
- БЕЗ преамбули («Зрозумів / Прийнято / Записав / Окей / Готово»)
- БЕЗ резюме і повторів
- Тільки конкретика, варіанти, або уточнююче питання

**Вихід:** Рома задає відкрите питання («чому / поясни / розкажи / порівняй / детально») або сам пише довгий контекст (>30 слів).

❌ «Зрозумів, ти питаєш який варіант. Подумавши над усіма трьома, схиляюсь до B тому що…»
✅ «B. Менше ризику, простіше відкотити.»

---

## 5. Без «Роби» — код не чіпати

Рома каже «Роби» — і тільки тоді змінюємо код. До цього — тільки обговорення, варіанти, читання.

ОДИН баг за раз. Після виправлення — пропоную наступний.

---

## 6. Глибина — читай код ПЕРЕД оцінкою

Не давати оцінку складності або план «на памʼять». Спершу читай реальний код залучених файлів, розумій залежності, тільки тоді відповідай.

Краще одна якісна відповідь ніж три поверхневі.

**Самоперевірка:** «Чи я читав код? Чи моя оцінка з фактів чи з припущень? Рома скаже "копни глибше"?»

---

## 7. Деплой коду → bump `CACHE_NAME` локально перед пушем

Зміни у `src/`, `index.html`, `style.css`, `sw.js` → змінити `CACHE_NAME` у `sw.js` (формат `nm-YYYYMMDD-HHMM`) **перед пушем**. CI його не чіпає.

Виняток: чисто документаційні зміни (`*.md` або `.claude/`) — `CACHE_NAME` НЕ чіпати.

---

## 8. Великі файли — skeleton + Edit, не один Write

Файли >250 рядків — спершу `Write` скелет з заглушками, потім кожен блок окремим `Edit`. Один великий `Write` 400+ рядків часто обриває stream timeout у Claude Code Web — втрачається прогрес.

Checkpoint-коміт після **кожної завершеної фази**, не чекати кінця.

---

## ✅ Самоперевірка перед першою реплікою

Після читання цього файлу `/start` змусить мене відповісти на 3 питання своїми словами в чаті. Це активна обробка — щоб правила не лишились пасивно прочитаними.
