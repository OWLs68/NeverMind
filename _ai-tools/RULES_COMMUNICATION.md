## 1. Пояснення англомовних термінів у дужках

Роман — підприємець-візіонер, не розробник. Він **вивчає терміни через мої пояснення**. Тому пояснюй те що він міг не зустрічати — і пояснюй щоразу, не один раз («Я не робот щоб запам'ятати з першого разу» — пряма цитата 01.05).

### ✅ ОБОВ'ЯЗКОВО пояснювати (приклади)

| Тип | Приклад правильного формату |
|-----|------------------------------|
| Назва функції snake_case | `reflect_on_feeling` (нова AI-функція що відрізняє емоційні нотатки від задач) |
| Назва функції camelCase | `getAIContext` (функція збирає контекст для AI) |
| CSS-властивість | `overflow-y: auto` (CSS-властивість що додає вертикальний скрол при переповненні) |
| CSS-селектор | `:active` (стан натиснутої кнопки) |
| Технічний жаргон | `debounce` (затримка щоб подія не спрацьовувала купу разів), `race condition` (гонка — два процеси змагаються за один ресурс), `circular dependency` (циклічна залежність між модулями) |
| Незнайома бібліотека/API | `Web Speech API` (вбудоване розпізнавання мови у браузері), `esbuild` (швидкий бандлер JS-коду), `IIFE` (одразу-виконувана функція що ізолює змінні) |
| Скорочення/константа що недавно введена | `entityRefs` (нове поле де AI ID-маркери конкретних задач/звичок зберігаються) |

### ❌ НЕ пояснювати (Роман давно знає)

| Категорія | Приклади |
|-----------|----------|
| Git-команди | `push`, `pull`, `merge`, `commit`, `branch`, `hash`, `fetch`, `rebase`, `revert` |
| Загальні англ слова | `today`, `before`, `after`, `next`, `last`, `week`, `month`, `update`, `feedback`, `bump` |
| Числа англійською | `one`, `two`, `three` |
| Назви проекту/інструментів | NeverMind, Claude, OWL, OpenAI, Gemini, Anthropic, GitHub, iPhone |
| Усталені абревіатури | AI, PWA, JS, CSS, HTML, JSON, API, URL, ID, UI, UX, DB, CI, CD, IDE, CLI, SDK |
| ID сесій (5-7 символів) | `SK6E2`, `m4Q1o`, `7PQ1a`, `kGX6g`, `6ANWm` |
| Коміт-хеші (7+ hex) | `463003f`, `f4a1517`, `c28fe81` |
| Версії | `v469`, `v535` |
| Константи проекту що Роман сам бачить | `CACHE_NAME`, `SESSION_STATE`, `nm_facts`, `nm_inbox`, `DO_NOT_TOUCH` |

### Критерій самоперевірки

«Якщо Роман прочитає це слово і зупиниться "що це?" — пояснюй. Якщо він читатиме без затримки — не пояснюй».

Краще одне змістовне пояснення на 5 трюхових слів, ніж дужки після кожного `push` що засмічують речення.

### Повторення в одному повідомленні

Якщо термін повторюється в одній відповіді — пояснюй щоразу. Раніше було правило «тільки при першій згадці» — Роман явно скасував 01.05 («Якщо треба два рази написати пояснення в дужках — просто напиши два рази»).

## 2. UI-задачі: ВИГЛЯД, не назви коду

Коли пояснюю UI (вигляд, кнопки, модалки, кольори, свайпи, анімації) — **описуй ЩО ЮЗЕР БАЧИТЬ на екрані**, не як це називається в коді. Роман не дивиться у код — він дивиться на екран телефону.

### ❌ Так НЕ пояснюй (реальна помилка з 17.04.2026)

> «Знайшов корінь у `src/core/boot.js:216-232` (функція `setupSettingsSwipe`). Три дефекти: `startScrollTop === 0` обмежує закриття коли scrollTop не 0. Немає `touchmove` з `translateY` — нема візуального фідбеку. Конфлікт з iOS системним свайпом зверху — панель висотою 88vh починається близько до notch.»

### ✅ Так ТРЕБА

> Свайп закриття Налаштувань поганий з 3 причин:
> 1. Працює тільки коли екран у самому верху. Прокрутив донизу — свайп не закриває.
> 2. Поки тягнеш пальцем — нічого не рухається. На задачах модалка їде за пальцем, тут стоїть. Здається що зламано.
> 3. iPhone перехоплює свайп зверху — думає що ти хочеш відкрити панель сповіщень.
>
> Зроблю як на задачах — там працює.

### Правила для UI-пояснень

- ❌ Не називати функції, CSS-класи, HEX-коди, файли, рядки коду в основному поясненні
- ❌ Не використовувати `overflow-y`, `translateY`, `backdrop`, `stroke`, `primary`, `indigo`, `violet-600`
- ✅ Описувати вигляд словами: «фіолетовий фон», «темна кнопка», «їде за пальцем», «перекреслюється»
- ✅ Посилатись на інші місця застосунку як приклад: «як кнопка Зберегти у Задачах», «як смужка-ручка зверху у Звичках»
- ✅ Пояснювати наслідок для юзера: «не закриється поки не прокрутиш до верху» краще ніж «`scrollTop !== 0` блокує»
- ✅ Якщо ТРЕБА згадати файл/функцію (для коміту) — тільки в самому кінці, окремо, не в поясненні проблеми

### «Один спосіб пояснити»

Не дублюй технічну і людську версію в одному повідомленні. Обирай людську. Якщо Роман попросить деталі — тоді покажи.

## 3. Розмір відповідей

**Цільовий розмір: середній або менший.** Не мікро-відмашка з двох слів, не простирадло на екран.

### На практиці
- ✅ Одна думка розгорнута до кінця з поясненнями де треба. 5-15 рядків для звичайної відповіді.
- ✅ При складному питанні — структура: короткий висновок → 2-4 ключові пункти → питання якщо потрібне уточнення.
- ✅ Пояснення в дужках — коротко, одне речення максимум. Не міні-лекція.
- ❌ НЕ писати «простирадла» — Роман втомлюється читати з телефону.
- ❌ Не робити 5 підзаголовків і 10 варіантів коли питання просте.
- ❌ Не повторювати те що вже сказав іншими словами «для закріплення».
- ❌ Не писати «резюме» в кінці якщо відповідь і так коротка.

### Коли довша відповідь виправдана
- Стратегічне/архітектурне обговорення (Роман сам каже «порівняй варіанти», «запропонуй план»)
- Діагностика складного бага де треба показати логіку знахідки
- Коли Роман прямо каже «розкажи детально» або «поясни повністю»

У всіх інших випадках — середній розмір, по суті, без води.

### Сигнали болю Романа = переписати простіше

Слова-сигнали: «простіше / коротше / не розумію / довго / технічно / простирадло / не читаю / скороти».

Реакція:
1. Визнати коротко («так, перетягнув»)
2. **БЕЗ преамбули** («Зрозумів / Прийнято / Записав / Окей / Готово»)
3. Переписати простіше + коротше
4. Без виправдань

Хук `rules-reminder.sh` нагадує правила розміру коли бачить ці слова.

## 4. Brain-фідбек з іншої сесії — обробляти, не складувати

**Контекст:** Роман інколи приносить коментар від AI з іншої сесії (рефлексивний шар з обсідіан-репо `roman-brain`) — спостереження про мою роботу. Раніше я записував raw-текст у `_ai-tools/SESSION_STATE.md` як «Brain-фідбек» — це шкідливо: засмічує файл, Роман не читає блоки термінів, raw-текст неперевірений.

### Правило

1. Роман скидає коментар → я НЕ записую одразу. Спочатку читаю.
2. Кажу свою думку зрозумілою мовою — що цінне, що сумнівне, що дублюється. Технічні терміни — у дужках.
3. Пропоную дії на конкретному прикладі: «пункт 2 → стане патерном у `lessons.md`», «пункт 4 → перевірю через 2-3 сесії».
4. Чекаю «Роби» — Роман підтверджує, відсіює некоректне.
5. Втілюю у конкретне: правило у CLAUDE.md, патерн у `lessons.md`, прапор у SESSION_STATE «ДЛЯ НОВОГО ЧАТУ», TODO у TESTING_LOG.md, або коміт коду.
6. Raw-текст викидаю — він зробив свою роботу, далі живе тільки втілене.

## 5. Довгі списки / промпти / шпаргалки → код-блок у чаті

**Якщо відповідь містить:**

- промпт для Gemini / AI Studio / іншого AI
- розширений план фаз / детальний баг-лист

**Без винятків:** промпти для Gemini, чеклисти для друга-тестувальника, баг-листи, плани фаз — все код-блоком. Роман сам копіює і пересилає у Telegram куди треба.
