## G-01 — Апостроф у `onclick` чіпів ламає клік

**Пріоритет:** 🔴 HIGH
**Файл:** `src/owl/board.js:137-140` (і потенційно `src/owl/inbox-board.js` — перевірити)
**Знайдено:** 2026-04-05, Gemini 3 Pro (аудит через скіл `/gemini`), підтверджено Claude Opus 4.6
**Статус:** open

### Опис
У `renderTabBoard()` чіпи рендеряться з інлайн `onclick` що містить текст чіпа:
```javascript
const s = escapeHtml(c).replace(/'/g, '&#39;');
return `<div class="owl-chip" onclick="owlChipToChat('${tab}','${s}')">${escapeHtml(c)}</div>`;
```

Спроба екранування апострофа через `&#39;` **не працює** — браузерний HTML-парсер декодує entity назад у `'` **перед** тим як JS-парсер побачить код атрибута.

### Чому це баг
Якщо текст чіпа містить апостроф (`'`), браузер генерує невалідний JS:
```
onclick="owlChipToChat('tasks','п'ять задач')"
                                  ↑
                         SyntaxError тут
```

**Конкретні кейси які гарантовано зламаються:**
- Українські слова з апострофом: **п'ять, м'ясо, сім'я, здоров'я, ім'я, об'єм, п'ятниця**
- Англійські фрази: **I'm, don't, can't**

AI (GPT-4o-mini) **може згенерувати** такий чіп — правила CLAUDE.md не забороняють апострофи. `sw.js`/PWA тут ні до чого, баг runtime при кліку.

### Пропоноване рішення
Перенести текст чіпа в `data-*` атрибути і повісити один делегований обробник:

```javascript
// Рендер — у renderTabBoard() або в _owlTabHTML():
const chipsHTML = (msg.chips || []).map(c =>
  `<div class="owl-chip" data-chip-tab="${escapeHtml(tab)}" data-chip-text="${escapeHtml(c)}">${escapeHtml(c)}</div>`
).join('');

// Один делегований обробник (в renderTabBoard, після присвоєння innerHTML):
chipsEl.addEventListener('click', (e) => {
  const chip = e.target.closest('.owl-chip');
  if (!chip) return;
  const t = chip.dataset.chipTab;
  const txt = chip.dataset.chipText;
  if (chip.classList.contains('owl-chip-speak')) {
    openChatBar(t);
  } else {
    owlChipToChat(t, txt);
  }
}, { once: false });
```

`dataset` безпечно зберігає будь-який текст — HTML-парсер не виконує його як JS.

**Перевірити також:** `src/owl/inbox-board.js` — можливо той самий паттерн використовується для чіпів Інбоксу.

### Спроби які не спрацювали
Ніякі — знайшли через `/gemini` аудит, ще не пробували фіксити.

### Тест-кейс ручний (після фіксу)
1. Відкрити DevTools Console на iPhone/Safari
2. У будь-яку вкладку табло підмінити чіп руками (або дочекатись згенерованого з апострофом)
3. Створити чіп з текстом `п'ять задач` через `saveTabBoardMsg('tasks', {text:'тест', chips:["п'ять задач"], ts:Date.now()})`
4. Тапнути на чіп
5. ✅ Має: відкрити чат-бар вкладки tasks з текстом "п'ять задач"
6. ❌ До фіксу: `SyntaxError: missing ) after argument list` у консолі, клік не спрацьовує

## G-02 — `chips.js` `includes()` дає false positives у навігації

**Пріоритет:** 🟡 MEDIUM (дизайн-smell, не критично поки AI дотримується правила "3 слова")
**Файл:** `src/owl/chips.js:26-33`
**Знайдено:** 2026-04-05, Gemini 3 Pro (аудит), підтверджено Claude
**Статус:** open

### Опис
`owlChipToChat()` розрізняє навігаційні чіпи від текстових через `String.prototype.includes()`:
```javascript
const navMap = [
  { patterns: ['задач', ...], tab: 'tasks' },
  ...
];
for (const nav of navMap) {
  if (nav.patterns.some(p => lower.includes(p))) {
    switchTab(nav.tab);
    return;
  }
}
```

### Чому це баг
Будь-який чіп що **містить** підрядок `"задач"` (або `"нотатк"`, `"фінанс"` тощо) перекине користувача на вкладку замість відправки в чат. Приклади хибних спрацювань:
- `"обговорити задачі"` — користувач очікує чат, отримує навігацію
- `"відкласти задачу на тиждень"` — порушує правило "3 слова" в промпті, але AI може згенерувати
- `"нотатки по проекту"` — теж хибний кидок на notes замість projects

**Поки працює** бо:
- Промпт до AI обмежує чіпи до 3 слів
- Реальні чіпи зазвичай імперативні: "закрити задачі", "відмітити звичку"

**Але фундаментально ненадійно** — покладається на дисципліну LLM.

### Пропоноване рішення
Явна типізація чіпів у відповіді AI — JSON-формат меняється:

**Було:**
```json
{"text": "...", "chips": ["закрити задачі", "пізніше"]}
```

**Треба:**
```json
{
  "text": "...",
  "chips": [
    {"label": "Закрити задачі", "action": "nav", "target": "tasks"},
    {"label": "Пізніше", "action": "chat", "text": "нагадай завтра"}
  ]
}
```

Тоді `owlChipToChat()` читає `chip.action` напряму, без парсингу тексту. Немає false positives взагалі.

### Що треба змінити при виправленні
1. Промпт у `src/owl/proactive.js` (`generateTabBoardMessage`) — оновити формат JSON і приклади
2. Промпт у `src/owl/inbox-board.js` (якщо там теж є проактивна генерація) — той самий
3. `saveTabBoardMsg()` — зберігати новий формат
4. `renderTabBoard()` у `board.js` — читати `chip.label` для відображення, передавати `action` в dataset
5. `owlChipToChat()` у `chips.js` — розгалуження по `action`, прибрати `navMap`
6. **Міграція:** старі збережені повідомлення у localStorage мають формат `chips:[string]`. Потрібен fallback — якщо `typeof chip === 'string'` → старий формат, обробляти як зараз (включно з includes), інакше новий.

## G-03 — `setTimeout(100)` у `chips.js` — тендітна синхронізація

**Пріоритет:** 🟢 LOW (смердить, але поки не ламається)
**Файл:** `src/owl/chips.js:43-52`
**Знайдено:** 2026-04-05, Gemini 3 Pro (аудит), частково підтверджено Claude
**Статус:** open

### Опис
Після виклику `openChatBar(barTab)` і підстановки тексту в input, відправка повідомлення робиться через жорсткий `setTimeout(..., 100)`:

```javascript
openChatBar(barTab);
const input = document.getElementById(inputId);
if (input) { input.value = text; ... }
setTimeout(() => { sendToAI(); /* або інша send-функція */ }, 100);
```

### Чому це (потенційно) баг
100 мс — магічне число без гарантій. На повільному iPhone з анімацією відкриття чат-бара або при великому списку повідомлень у `restoreChatUI()` ця затримка може не покрити час ініціалізації.

**Примітка Claude:** Gemini помилково описав що `getElementById` повертає null — насправді він викликається синхронно до setTimeout, і в NeverMind input'и живуть у статичному `index.html`. Тому проблеми з null немає. Реальна потенційна проблема — race з пересіканням рендерів OWL.

### Пропоноване рішення
Зробити `openChatBar()` асинхронним і повертати Promise який резолвиться після повного відкриття:

```javascript
// в ai/core.js
export async function openChatBar(tab) {
  // ... існуюча логіка
  await new Promise(r => requestAnimationFrame(r)); // або waitForAnimationEnd
  return;
}

// в chips.js
await openChatBar(barTab);
// ... всі маніпуляції вже без setTimeout
sendToAI();
```

### Важливо — коли робити
**Тільки** в рамках окремого рефакторингу чат-бара. Не чіпати ізольовано — `openChatBar` викликається з багатьох місць, зміна сигнатури на async потребує оновлення всіх caller'ів і тестування кожного.

## Wontfix / Дискусія

### Галюцинація Gemini: "синтаксична помилка `<msg.id>`" у `board.js:125`
**Статус:** wontfix — не баг

Gemini під час того ж аудиту повідомив про "критичну синтаксичну помилку" у конструкції:
```javascript
const ago = Date.now() - (msg.ts || msg.id || Date.now());
```
Стверджуючи що там є `<msg.id>` з кутовими дужками. **Це галюцинація** — у реальному коді дужок немає, це валідний JS-ланцюжок з `||` fallback.

**Записано як урок:** завжди перевіряти точну цитату Gemini проти реального коду перед виправленням. Не довіряти "критичним синтаксичним помилкам" без верифікації.
