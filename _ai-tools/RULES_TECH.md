# RULES_TECH — технічні правила (код, AI tools, рефакторинг, деплой, документація)

> Читай при технічних задачах: новий JS-файл, AI tool, рефакторинг великого файлу, деплой, оновлення документації.

## 1. Робочий процес для надійності проти обривів

> Додано 17.04.2026 (gHCOh) після рефакторингу `finance.js`. Stream idle timeout (обрив зв'язку через бездіяльність потоку) обірвав відповіді 5+ разів.

### Правило 1.1 — Великі файли через skeleton+Edit

**Тригер:** створюєш JS/HTML-файл де попередній Write буде >250 рядків.

**Дія:**
1. Write скелет з заглушками — імпорти + `export function foo() {}` плейсхолдери + window-exports (~15-30 рядків)
2. Для кожного логічного блоку — окремий Edit який замінює заглушку реалізацією
3. Після всіх Edit — `build` перевірка
4. При обриві посеред — продовжую з останньої заглушки, попередні блоки не постраждали

**Чому:** один великий Write 400+ рядків у Claude Code Web часто обриває stream idle timeout. Edit-блоки по 50-100 рядків майже завжди проходять бо коротші.

**Виняток:** файли <250 рядків — одразу Write, без надобудов.

### Правило 1.2 — Checkpoint-коміти після кожної фази

**Тригер:** багатофазна робота (рефакторинг, міграція, великий фікс у кілька підсистем).

**Дія:** після кожної завершеної логічної підзадачі — коміт + push. Не чекати кінця всіх фаз.

**Приклад з рефакторингу `finance.js` (gHCOh):** 6 фаз по винесенню модулів. Після Фази 1 одразу коміт → Фази 2-4 → коміт → тоді посередині Фази 5 обрив не втратив би Фази 1-4 (вони вже на GitHub).

### Правило 1.3 — Мінімум тексту між tool calls

**Тригер:** послідовність tool calls під час виконання задачі.

**Дія:** між tool calls писати **одне речення-максимум** (<25 слів). Описові абзаци економлять увагу юзера і зменшують ризик обриву.

**Виняток:** перший і останній меседж у задачі — тут можна ширше (план на старті + підсумок у кінці).

1. **Реалізація** у `src/ai/prompts.js` (INBOX_TOOLS) або `src/ai/ui-tools.js` (UI_TOOLS)

**Хук `ai-tools-sync.sh`** автоматично нагадує оновити `docs/AI_TOOLS.md` коли я редагую `prompts.js`/`ui-tools.js`.

## 3. Деплой (CACHE_NAME bump)

### При деплої КОДУ

**Тригер:** зміни у `src/`, `index.html`, `style.css`, `sw.js`.

**Дія перед пушем:**
1. **Локально** змінити `CACHE_NAME` у `sw.js` (формат: `nm-YYYYMMDD-HHMM`, наприклад `nm-20260501-1430`).
2. Використовуй `date` в терміналі щоб не вигадувати: `date +nm-%Y%m%d-%H%M`.
3. CI **не чіпає** `sw.js` — це треба робити вручну.

**Виняток:** чисто документаційні зміни (тільки `*.md` файли або `.claude/` скіли) — `CACHE_NAME` **НЕ чіпати**, кеш PWA цих файлів не торкається.

### При кожному пуші

- НЕ вигадувати час на бейджі (бейдж — напис «Версія N · DD.MM HH:MM» у застосунку). Бейдж встановлює CI автоматично.
- Цей час потрібен Роману щоб перевірити чи застосунок оновився на телефоні — «старий час = стара версія, новий час = деплой проїхав».

### Якщо деплой не спрацював

```bash
git commit --allow-empty -m "ci: retrigger" && git push origin <branch>
```

(Порожній коміт який заново запускає CI без зміни коду.)

### Деталі системи деплою

`docs/TECHNICAL_REFERENCE.md` → секція «Система деплою» (один job, `-X theirs`, concurrency, cache-bust правила).

## 4. i18n — обгортки `t('key', 'fallback')` для нових user-facing рядків

**Контекст:** `t(key, fallback, params)` у `src/core/utils.js` — заглушка для майбутньої локалізації (Польща, Німеччина у планах). Зараз повертає український fallback, кожен новий текст має «мітку» (key) яку потім використає словник перекладу.

**Параметри:** `t('greeting', 'Привіт, {name}!', { name: 'Роман' })` → `"Привіт, Роман!"`.

**Приклад:** замість `'Немає нових задач'` → `t('inbox_empty', 'Немає нових задач')`.

### Автоматизація

Правило контролюється **CI-скриптом + PostToolUse-хуком** (29.04 m4Q1o):

1. **Білд ламається** якщо у будь-якому файлі `src/` зросла кількість необгорнутих українських рядків. Скрипт: `scripts/check-i18n.js` (запускається з `build.js` перед esbuild). Baseline: `i18n-baseline.json` — поточно 1426 рядків у 25 файлах.
2. **PostToolUse хук** `.claude/hooks/i18n-reminder.sh` показує кількість і приклади необгорнутих рядків коли я правлю файл у `src/`.
3. **Whitelist:** `src/ai/*` (AI-промпти лишаються українськими), `src/owl/*` (mixed AI+UI), виклики `t(...)`, `console.{log,error,warn,info}(...)`, `toLocaleDateString/String/TimeString(...)`, коментарі.
4. **Workflow:** після обгортання N рядків → `node scripts/check-i18n.js --update-baseline` → коміт з оновленим `i18n-baseline.json`.
5. **Екстрений деплой без перевірки:** `SKIP_I18N_CHECK=1 node build.js` (не зловживати).

## 5. Аудит скілів перед створенням або використанням

**Тригер:** перед тим як створити новий скіл у `.claude/commands/` або виконати наявний.

### Self-audit за 5 пунктами

1. Чи скіл торкається пунктів з `docs/DO_NOT_TOUCH.md`?
2. Чи DOM-селектори/CSS-класи/функції які скіл використовує — **реально існують** у коді? (Якщо ні — скіл пише код у порожнечу)
3. Чи скіл дублює існуючу архітектуру (OWL модулі, Trash, tabs, boot)? Якщо так — інтегруй, не створюй паралельну.
4. Чи скіл нагадує про `CACHE_NAME` bump при зміні коду (`index.html` / `style.css` / JS-файли)?
5. Чи скіл делегує на `/new-file` при створенні нових JS-модулів у `src/`?

**Якщо знайдено порушення — зупинись і повідом Роману список. Не виконуй скіл самочинно.**

**Чому додано (16.04.2026 W6MDn):** інша сесія написала скіли `/owl-motion` і `/pwa-ios-fix` з DOM-селекторами яких у нас немає (`#owl-container`, `.tab-content`) — потребувало ручного виправлення.

## 6. Усі скіли і хуки — у репо `.claude/`, не у `~/.claude/`

**Контекст (29.04.2026 m4Q1o):** папка проекту `.claude/` (скіли у `commands/`, хуки у `hooks/`, налаштування у `settings.json`) — частина git-репо, видна на GitHub, переноситься між акаунтами одним клонуванням. Папка `~/.claude/` у домашній директорії — приватна для одного акаунту Claude Code, **зникає при переході на інший акаунт**.

### Правило

Будь-який новий скіл / хук / зміна `settings.json` створюється **виключно у `.claude/`** (відносно кореня проекту), потім додається у git. Заборонено зберігати у `~/.claude/` навіть тимчасово.

**Виняток:** `.claude/settings.local.json` (особисті преференції одного акаунту, у `.gitignore`) — туди НІЧОГО важливого, тільки тимчасові експерименти.

Якщо знайду в `~/.claude/` щось що повинно бути в репо — попередити Романа і запропонувати перенести.

## 7. `safeAgentReply` — завжди замість прямого `addMsg('agent', reply)`

`safeAgentReply` — обгортка яка перевіряє чи відповідь агента не є сирим JSON (сирий JSON — коли AI повернув службову технічну відповідь замість людського тексту). Без обгортки юзер бачить `{"action":"save_task",...}` замість живої відповіді.

→ Священна корова. Не обходити.

## 10. Збірка

`node build.js` → `src/app.js` → `bundle.js` (esbuild, IIFE формат — одразу-виконувана функція що ізолює змінні).

**Порядок імпортів в `src/app.js` критичний** — відповідає порядку оригінальних `<script>` тегів. AI-модулі раніше за табси для уникнення circular dependencies. Деталі → `docs/TECHNICAL_REFERENCE.md` → «Міжмодульні залежності».
